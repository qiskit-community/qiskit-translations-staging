Участие в разработке Qiskit¶

Qiskit - проект с открытым исходным кодом цель которого - дать людям возможность использовать квантовые вычисления. На этой странице Вы найдете информацию о том, как присоединиться к сообществу Qiskit, чтобы помочь достижению этой цели.

Исходный код¶

Исходный код Qiskit находится в репозиториях Qiskit на GitHub, где Вы найдете отдельные проекты, из которых состоит Qiskit:

  • Qiskit Terra

  • Qiskit Aer

  • Qiskit Ignis

  • Qiskit Aqua

  • Qiskit IBMQ Provider

  • Документация по Qiskit API

С чего начать¶

Узнайте, как члены сообщества Qiskit

  • Связаны друг с другом

  • Обсуждают идеи

  • Помогают друг другу решать проблемы

  • Остаются в курсе новостей сообщества

  • Cохраняют единый согласованный стиль кода

  • Устанавливают пакеты Qiskit из исходного кода

Участие в работе над кодом¶

Руководство по стилю¶

Для обеспечения единого согласованного стиля кода в проекте, мы используем Pylint и pycodesytle чтобы убедиться, что Ваш вклад соответствует установленному стилю проекта. Для того, чтобы проверить Ваши изменения на соответствие стилю, выполните команду: tox -elint

Лицензионное соглашение участника¶

Прежде чем Вы сможете добавить новый код, Вы, как участник, должны подписать лицензионное соглашение (CLA). Подписав CLA, Вы подтверждаете, что являетесь автором данного кода и свободно вносите свой вклад в соответствии с условиями Apache-2.0 лицензии.

Когда Вы вносите свой вклад в любой из проектов Qiskit и создаете новый запрос на слияние (Pull Request), бот проверяет, подписали ли Вы CLA. При необходимости бот оставит комментарий и предоставит ссылку для принятия соглашения. Также CLA доступен для просмотра в формате PDF.

Примечание

Если Ваш вклад является частью Вашей трудовой деятельности или Ваш код является собственностью Вашего работодателя, то скорее всего Вам также нужно будет подписать корпоративный вариант CLA и отправить нам его на почту <qiskit@us.ibm.com>.

Запросы на слияние (Pull Request)¶

Мы используем ` GitHub pull requests <https://help.github.com/articles/about-pull-requests>` __ для приема вкладов.

Хотя это и не обязательно, но открытие новой задачи (issue) с сообщением о том, что вы исправляете ошибку или разрабатываете новую функцию, прежде чем вы откроете pull request, является важным шагом для начала обсуждения своей работы с сообществом. Открытая задача предоставляет пространство для обсуждения идеи и возможностей совместной работы по ее реализации в коде. Она также позволяет сообществу понять, над чем вы работаете, и дает возможность ссылаться на нее при обсуждении с другими членами сообщества и команды.

Если вы написали часть решения, но вам нужна помощь в его завершении, или вам нужно получить обратную связь перед окончанием разработки, или же вы просто хотите поделиться им и обсудить, вы можете открыть Pull request с пометкой В работе. При создании Pull Request необходимо добавить в заголовок тег [WIP] (Work in Progress). Он укажет, что код в Pull request не закончен и может измениться. Также он означает то, что коммиты не будут объединены с целевой веткой до тех пор, пока работа не будет завершена. После того, как код полностью готов к проверке для и слиянию, можно удалить тег [WIP].

Комментарии к коммитам¶

Содержание комментария в коммите, описывающего изменения, так же важно, как и само изменение. Комментарий коммита предоставляет контекст не только для проверки кода, но и для истории изменений в git. Подробный комментарий к коммиту сделает его проще для проверки, а также предоставит контекст сделанных Вами изменений, когда кто-то смотрит на них в будущем. При написании комментарию к коммиту, помните следующие важные детали:

Не предполагайте, что проверяющий понимает, какая была исходная проблема.

При чтении содержания задачи, после ряда комментариев, часто становится ясно, что является причиной проблемы. В комментарии к коммиту должно быть четкое отражение того, в чем состоит первоначальная проблема. При исправлении ошибки часто интересна историческая информация о том, как выявлена проблема. Должно быть возможно проверить предложенное изменение кода с помощью комментария к коммиту, без необходимости читать все комментарии к задаче.

Не предполагайте, что код является очевидным/авто-документированным.

Что очевидно для одного человека, может быть непонятным для другого. Всегда документируйте то, в чем состояла изначальная проблема и как она исправлена. Делайте это для любых изменений, за исключением наиболее очевидных, таких как опечатки.

Описывайте, почему сделаны данные изменения.

Распространенная ошибка - задокументировать написанный код решения, не описав, почему разработчик решил это сделать. Разумеется Вы должны описать общую структуру кода, особенно для больших изменений, но, что более важно, не забудьте описать что побудило Вас сделать эти изменения.

Перечитайте комментарий к коммиту, чтобы проверить, помогает ли он понять улучшения структуры кода.

Часто при написании большого комментария к коммиту становится очевидно, что он должен быть разделен на два или более. Не бойтесь возвращаться назад и разделять изменения, чтобы включить их в разные Pull request.

Убедитесь, что информации достаточно, чтобы решить, следует ли проверять код.

Когда GitHub отправляет оповещения по электронной почте о новых Pull request, в них содержится минимальная информация - обычно только комментарий к коммиту и список изменений в файлах. Из-за большого объема, комментарий должен содержать достаточную информацию, чтобы потенциальные проверяющие смогли найти правку, которую им нужно рассмотреть.

Первая строка комментария к коммиту - самая важная.

В коммитах Git особое значение имеет первая строка комментария. Она по умолчанию используется в качестве заголовка Pull Request, темы письма в email уведомлениях, git-аннотациях, а также в качестве комментария к коммиту слияния (merge commit) и много других мест, где краткость имеет большое значение. Помимо обобщения самих изменений, она должна отражать, какую часть кодовой базы затрагивают изменения.

Кроме того, первая строка комментария к коммиту становится записью в автоматически генерируемом журнале изменений, если Pull request отмечен как включаемый в него. Критически важно, чтобы Вы писали четкие и емкие комментарии.

Опишите любые ограничения текущего кода.

Если измененный код имеет задачи для будущих улучшений или известные ограничения - упоминайте их в комментарии к коммиту. Это демонстрирует проверяющему то, что была рассмотрена широкая картина, а также компромиссы, на которые пришлось пойти для достижения краткосрочных целей и долгосрочные возможностей для улучшения.

Включайте ссылки на задачи.

Если исправления в коммите связаны с задачей (issue), убедитесь, что Вы упомянули ее в комментарии к коммиту. Используйте синтаксис:

Fixes #1234

если они закрывают задачу (GitHub закроет ее, когда Pull request будет одобрен и изменение добавлено в целевую ветку).

Основное правило, которому стоит следовать:

Комментарий к коммиту должен содержать всю информацию, необходимую для полного понимания и проверки изменений на корректность.

Документирование кода¶

При изменении элемента необходимо обновить связанную с ним документацию и ее части из раздела docs/apidocs в соответствующем репозитории. Для того, чтобы создать документацию локально, выполните команду tox -edocs, которая создаст и скомпонует документацию локально, а затем сохранит результат в docs/_build/html. Дополнительно Docs CI также запустит эту команду и сохранит zip-файл с сгенерированной документацией, который можно будет позже загрузить и просмотреть локально.

Если у Вас есть проблема с документацией, которая поддерживается в репозитории Qiskit/qiskit, то Вы можете открыть задачу с типом Documentation. Например, Вы заметили баг, или у Вас есть новая функция, которую нужно задокументировать, или же у Вас есть материал, который можно добавить в существующие документы.

Первый вклад¶

Если вы хотите внести свой вклад в Qiskit, но не уверены, с чего начать, метка «good first issue» ставится на задачи, подходящие для новых людей на проекте. Эти задачи были рассмотрены и отмечены другими участниками как задачи, над которыми сможет работать новый участник. Другими словами, близкое знакомство с Qiskit не является необходимым условием для решения таких задач.

Политика устаревания¶

Пользователи Qiskit должны знать, будет ли поддерживаться та или иная функция или API в ближайшем будущем. Также важно для пользователя и знание, при каких условиях в проекте может удаляться (или изменять обратную совместимость) функция или API. Чтобы вы могли это понимать, следующая политика отражает, как происходит работа при устаревании/удалении компонентов в Qiskit:

1. Features, APIs, or configuration options are marked deprecated in the code. Appropriate DeprecationWarning class warnings will be sent to the user. The deprecated code will be frozen and receive only minimal maintenance (just so that it continues to work as-is).

2. A migration path will be documented for current users of the feature. This will be outlined in the both the release notes adding the deprecation, and the release notes removing the feature at the completion of the deprecation cycle. If feasible, the warning message will also include the migration path. A migration path might be «stop using that feature», but in such cases it is necessary to first judge how widely used and/or important the feature is to users, in order to determine a reasonable obsolescence date.

3. An obsolescence date for the feature will be set. The feature must remain intact and working (although with the proper warning being emitted) in all releases pushed until after that obsolescence date. At the very minimum, the feature (or API, or configuration option) should be marked as deprecated (and continue to be supported) for at least three months of linear time from the release date of the first release to include the deprecation warning. For example, if a feature were deprecated in the 0.9.0 release of Terra, which was released on August 22, 2019, then that feature should still appear in all releases until at least November 22, 2019. Since releases do not occur at fixed time intervals, a deprecation warning may only occur in one release prior to removal.

Стоит отметить, что этот интервал является минимальным. Для более значительных функций рекомендуется хотя бы удвоить это время. Также, согласно политике стабильности ветки, удаление устаревших функций может происходить только в минорных версиях (версии без значительных изменений). Также, они не могут использоваться при портировании новых функций для поддержки в более старых версиях (backporting).

Предупреждения об устаревании (Deprecation Warnings)¶

Корректным способом предупредить об устаревании является использование функции warn из модуля предупреждений (warnings module) в стандартной библиотеке Python. В качестве категории предупреждения следует использовать класс DeprecationWarning. Пример использования:

import warnings

def foo(input):
    warnings.warn('The qiskit.foo() function is deprecated as of 0.9.0, and '
                  'will be removed no earlier than 3 months after that '
                  'release date. You should use the qiskit.bar() function '
                  'instead.', DeprecationWarning, stacklevel=2)

Также необходимо пояснить использование параметра stack_level при вызове функции warn(). Этот аргумент используется для указания, какой уровень в стеке вызовов будет использоваться в качестве инициализации предупреждения. Обычно значение stack_level должно быть равно 2, так как в этом случае будет отображена строка вызова контекста, в котором было вызвано предупреждение. В приведенном выше примере это будет объект, вызвавший метод foo(). Если же такое значение не будет установлено, то причиной отображения предупреждения будет строка внутри функции foo(), что не сможет помочь пользователю, пытающемуся определить, откуда производится вызов устаревшего метода. Однако, это значение может быть изменено, в зависимости от стека вызовов и места, откуда вызывается функция warn(). Например, если функция предупреждения всегда вызывается приватным методом, который может вызвать только экземпляр одного класса, то целесообразно выбрать stack_level=3.

Политика стабильной ветки¶

Стабильная ветка призвана быть безопасным источником исходного кода, содержащего исправления серьезных ошибок и проблем с безопасностью, которые были исправлены в master-ветке с момента релиза. При просмотре PR с целевой стабильной веткой, мы обязаны держать баланс между риском, который могут принести изменения, и пользой, которую данные изменения несут пользователям. Только ограниченный класс изменений подходит для включения в стабильную ветку. Например, большое изменение для серьезной проблемы, или же простое исправление для сложновоспроизводимой проблемы. При рассмотрении изменения следует учесть ряд факторов:

  • Риск регрессии: даже самые мелкие изменения сопряжены с определенным риском того, что что-то перестанет работать, и мы хотим избегать таких регрессий на стабильной ветке.

  • Видимость для пользователя: исправляем ли мы то, что пользователи могут на самом деле заметить, и если да, то насколько это важно?

  • Насколько самодостаточны исправления: если они исправляют значительную проблему, а также требуют рефакторинга крупных частей кодовой базы, то возможно стоит подумать о том, как можно сделать исправление менее рискованным.

  • Наличие исправления в мастер-ветке: если изменение не имеет значения для основной ветки (master), это изменение должно быть резервным портом того, которое уже было объединено с master.

Процедура бэкпорта (Backporting):¶

При бэкпортинге (применении изменений, предназначенных для основной, развивающейся в данный момент версии программы, к более старым версиям) из master-ветки в stable-ветку, мы хотим сохранять ссылку на изменение в master. При создании ветки для запроса на слияние (Pull request) в stable-ветку используйте:

$ git cherry-pick -x $master_commit_id

Однако это работает только для небольших патчей от master-ветки. Если Вам нужно выполнить обратное портирование нескольких более крупных коммитов (например, из PR со слитыми (squashed) коммитами) от master-ветки, сделайте это вручную. В этих случаях следует добавить:

Backported from: #master pr number

так мы можем отслеживать источник изменений, даже если cherry-pick не даст приемлемых результатов.

Если предложенные Вами изменения не будут проходить cherry-pick автоматически, Вы можете помочь, разрешив конфликты самостоятельно и предложив результирующие изменения. Пожалуйста, оставляйте конфликтные строки в комментарии к коммиту, чтобы помочь проверить стабильный результирующий патч.

Лэйблы бэкпортов¶

Ошибки или PR, помеченные stable backport potential, — это ошибки, которые также применяются к стабильному выпуску и могут быть пригодны для бэкпортинга после исправления их в master-ветке. Как только бэкпорт будет предложен, этот тэг должен быть удален.

Добавляйте [Stable] в заголовок PR с целевой stable-веткой, как знак того, что установка этой ветки в качестве целевой не была ошибкой. Также, добавьте ссылку на номер PR с целевой master-веткой, которую Вы портируете.

Участие в работе над документацией¶

Документация Qiskit формируется в философии docs as code, и составляется прежде всего из комментариев кода Qiskit в стиле документации к API.

Актуальная документация собирается из master-ветки Qiskit/qiskit/docs при помощи Sphinx. Большая часть документации из раздела API Reference взята из комментариев кода в репозиториях, перечисленных в Исходный код.

Структура документации¶

Метод структурирования документации в Qiskit заключается в том, чтобы задокументировать как можно больше. Это облегчает внесение дополнений и исправлений во время разработки, так как большинство документации находится рядом с изменяемым кодом. В структуре документации проекта Terra есть три уровня:

.rst файлы в docs/apidocs

Эти файлы используются, чтобы указать Sphinx, какие модули следует включать в формирование документации. Они содержат два вида информации: внутренние ссылки или перекрестные ссылки к модулю, который может использоваться для внутренних ссылок внутри документации, а также директива automodule, которая используется для парсинга модулей из указанного пути их импорта. Например, файл dagcircuit.rst содержит:

.. _qiskit-dagcircuit:


.. automodule:: qiskit.dagcircuit
   :no-members:
   :no-inherited-members:
   :no-special-members:

Единственный .rst файл вне данной папки - это qiskit.rst, который содержит оглавление (table of contents). Если Вы добавляете новый .rst файл для документирования нового модуля, не забудьте добавить его в toctree в этом файле.

Документация на уровне модуля

Эта документация находится на уровне того модуля, который указан в директиве automodule rst файла. Если указанный модуль является каталогом/пространством имен (namespace), то документация для этого каталога должна находиться в файле __init__.py. Документация на уровне модуля содержит подробную информацию о нем. Обычно в данном виде документации описываются все классы и функции API, содержащиеся в этом модуле. Это делается с помощью autosummary directive (или autodoc directives напрямую, если модуль простой, например, в случае qiskit.execute). Директива autosummary предназначена для автоматического документирования нескольких различных элементов Python (классы, функции и т.д.) без необходимости вручную вызывать инструменты автодокументирования для каждого из них. Документация на уровне модуля является высокоуровневым обзором функциональности, которую предоставляет модуль вцелом. Обычно это делается путем группировки различных компонентов API в несколько подразделов.

Например, документация модуля dagcircuit в qiskit/dagcircuit/__init__.py будет содержать следующее:

"""
=======================================
DAG Circuits (:mod:`qiskit.dagcircuit`)
=======================================
.. currentmodule:: qiskit.dagcircuit
DAG Circuits
============
.. autosummary::
   :toctree: ../stubs/
   DAGCircuit
   DAGNode
Exceptions
==========
.. autosummary::
   :toctree: ../stubs/
   DAGCircuitError
"""

Примечание

Данный пример может отличаться от той документации, которую фактически содержит модуль dagcircuit.

Настоящая документация для перечисленных элементов представлена в модуле docstring

Необходимо тщательно документировать все публичные интерфейсы, приводя при необходимости примеры использования. Для документации используется стиль Google Python Style Docstrings. Парсинг производится при помощи расширения napoleon sphinx. Документация napoleon sphinx содержит хорошие примеры того, как должна быть сформирована документация.

Примечание

Вы можете использовать любую из директив Sphinx или rst форматирование в документации. Например, одно из распространенных расширений — jupyter-execute, которое используется для выполнения блока кода в Jupyter и отображения кода, вместе с результатом его исполнения. Это особенно полезно для визуализации результатов.

Интеграция документации¶

Размещенная на сайте https://qiskit.org/documentation/ документация охватывает весь проект Qiskit; Terra является всего лишь одним из его компонентов. Сборки документации для актуальной версии фреймворка собраны репозиторием Qiskit meta-package https://github.com/Qiskit/qiskit. Когда коммиты попадают в данный репозиторий, сборки Sphinx загружаются на сайт qiskit.org. Эти сборки сконфигурированы так, чтобы получать документацию тех версий элементов Qiskit, которые установлены мета-пакетом. Например, если текущая версия meta-пакета 0.13.0, то будет скопирована документация из релиза Terra 0.10.0. Когда требования метапакета изменяются, начинает подгружаться документация уже новой версии. Это означает, что исправления ошибочной документации API должны также быть включены в новый релиз. Исправления документации - это допустимые бэкпорты для выпуска патча в stable-ветку соответствии с политикой стабильной ветки (см. этот раздел ниже).

В процессе сборки, содержимое каждого элемента в docs/apidocs/ рекурсивно копируется в общедоступную копию doc/apidocs/ в репозиторий мета-пакетов вместе со всеми остальными элементами. Это означает, что все, что находится в корне docs/apidocs каждого элемента в релизе будет в корне https://qiskit.org/documentation/apidoc/.

Перевод документации¶

Документация по Qiskit переведена с помощью платформы Crowdin - веб-платформы локализации, которая позволяет организациям координировать работу над переводами и сотрудничать с сообществами с целью перевода материалов. Crowdin позволяет нашему сообществу переводчиков увеличивать их вклад, автоматически переиспользуя уже сделанную работу, используя ранее переведенные предложения для помощи с переводом других похожих предложений. Crowdin также делает переводы устойчивыми ко многим типам изменений оригинального материала, таким как перемещение предложений, даже между разными файлами.

Запросы на перевод материалов Qiskit обрабатываются в репозитории ` Qiskit Translations <https://github.com/Qiskit/qiskit-translations>` __. Для того, чтобы внести вклад в локализацию Qiskit, Вам нужно сделать следующее:

  1. Добавить свое имя (или ID) в файл LOCALIZATION_CONTRIBUTORS.

  2. Создать Pull request (PR) для слияния изменений. Убедитесь, что ваш Pull request соответствует шаблону.

    Примечание

    • Каждый участник должен создать свой собственный PR и подписать CLA.

    • Пожалуйста, укажите в PR язык, на котором Вы хотели бы внести свой вклад.

    • Если есть открытая задача по добавлению языка в проект, добавьте ссылку на нее в PR.

  3. Вам будет предложено подписать Лицензионное соглашение участников проекта Qiskit Contributors (CLA); пожалуйста, сделайте это.

  4. Для новых языков необходимо как минимум три участника для того, чтобы он получил официальную поддержку от администраторов проекта локализации и был добавлен.

  5. Среди участников проекта следует выделить руководителя перевода, который будет служить связующим звеном с администраторами проекта локализации. Руководитель должен связаться с Yuri Kobayashi (yurik@jp.ibm.com) по электронной почте.

  6. В Crowdin проекте Qiskit-Docs выберете язык, с переводом на который Вы хотите работать.

    Примечание

    Как уже упоминалось в блог посте Qiskit in my language is Qiskit, мы хотим убедиться, что языки имеют достаточную поддержку для создания команды переводчиков, корректоров и руководителей команд. Если Вы хотите быть руководителем команды переводчиков или же хотите вступить в команду, Вы можете открыть задачу на GitHub, чтобы начать обсуждение с командой Qiskit и найти участников для работы над переводом.

  7. Нажмите на кнопку Join и paste the URL of your PR в диалоговом окне, где Вас спросят, почему Вы хотите присоединиться к проекту Crowdin.

Администраторы проекта Crowdin рассмотрят Ваш запрос и предоставят Вам доступ как можно скорее.

Сборка из исходного кода¶

Вы можете создать локальную копию документации из своей копии репозитория Qiskit/qiskit следующим образом:

  1. Склонируйте репозиторий Qiskit.

    git clone https://github.com/Qiskit/qiskit.git

  2. Клонирование репозитория создает локальную папку с названием``qiskit``.

    cd qiskit

  3. Создайте документацию, открыв Вашу локальную копию Qiskit/qiskit в терминале и выполнив следующую команду.

    tox -edocs

    Если у Вас еще не установлен инструмент для выполнения команды tox, установите его, выполнив команду:

    pip install tox

При внесении изменений в локальные файлы RST, Вы можете обновить Ваши HTML-файлы, перейдя в директорию /doc/ и выполнив команду в терминале:

tox -edocs

Она создаст стилизованную HTML-версию Вашего локального репозитория документации в каталоге /docs/_build/html/.

Установка из исходного кода¶

Установка элементов из исходного кода позволит Вам получить доступ к последней обновленной версии Qiskit вместо использования версии из репозитория Python Package Index (PyPI). Это даст Вам возможность более эффективно изучить и улучшить последнюю версию Qiskit кода.

При установке компонентов из исходного кода, по умолчанию будет использоваться их development версия (которая соответствует master ветке в git), а не stable версия (которая содержит тот же код, что и опубликованные пакеты pip). Поскольку development версии элемента или компонента обычно включают в себя новые функции и последние изменения, они обычно требуют использования development версии и остальных компонентов.

Примечание

Пакеты Terra и Aer требуют компилятор для сборки из исходного кода перед установкой. Ignis, Aqua и IBM Quantum Provider не требуют компилятора.

Установка элементов из исходного кода требует следующего порядка установки. Это нужно для предотвращения использования версий, которые могут быть ниже желаемых, если pip версия старее, чем версия из исходного кода:

  1. qiskit-terra

  2. qiskit-aer

  3. qiskit-ignis

  4. qiskit-aqua

  5. qiskit-ibmq-provider (если вы хотите подключаться к устройствам IBM Quantum или онлайн-симулятору)

Для одновременной работы с несколькими компонентами используйте следующие шаги для каждого элемента.

Примечание

Из-за использования пространства имен в Python, необходимо следить за тем, как устанавливаются пакеты. Не используйте метапакет qiskit, если Вы планируете установить какой-либо элемент из исходного кода. Также следуйте данному руководству и используйте отдельную виртуальную среду (virtual environment) для разработки. Если Вы хотите смешать существующие установленные пакеты с Вашими пакетами для разработки, перейдите на страницу https://github.com/pypa/sample-namespace-packages/blob/master/table.md, чтобы найти рабочие комбинации.

Настройка виртуальной среды разработки¶

conda create -y -n QiskitDevenv python=3
conda activate QiskitDevenv

Установка пакета Terra из исходного кода¶

Установка из исходного кода требует наличия компилятора C++ в системе, поддерживающей C++11.

На большинстве Linux-платформ уже установлен необходимый компилятор GCC.

Если Вы используете macOS, то можно установить компилятор Clang, предварительно установив XCode. Проверьте, установлены ли XCode и Clang на Вашей системе, открыв окно терминала и введя следующее.

clang --version

Установите XCode и Clang с помощью следующей команды.

xcode-select --install

В Windows проще всего установить компилятор Visual C++ из Build Tools for Visual Studio 2017. Также Вы можете установить Visual Studio версий 2015 или 2017, выбрав в параметрах установки компилятор С++.

После установки компилятора все готово для установки Qiskit Terra.

  1. Склонируйте репозиторий Terra.

    git clone https://github.com/Qiskit/qiskit-terra.git

  2. Клонирование репозитория создает локальную папку с названием qiskit-terra.

    cd qiskit-terra

  3. Установите необходимые библиотеки Python из Вашей папки qiskit-terra.

    pip install cython

  4. Если Вы хотите запустить тесты, установите необходимые инструменты для разработчика.

    pip install -r requirements-dev.txt

  5. Установите qiskit-terra.

    pip install .

Если Вы хотите установить пакет в редактируемом режиме, который позволяет не переустанавливать пакет в случае изменения кода в проекте, Вы можете сделать это с помощью:

pip install -e .

После установки Terra, Вы можете запустить примеры работы с кодом. Запуск примера производится следующей командой.

python examples/python/using_qiskit_terra_level_0.py

Примечание

Если Вы не хотите устанавливать какие-либо другие компоненты, qiskit-terra выведет RuntimeWarning - предупреждение о том, что qiskit-aer и qiskit-ibmq-provider не установлены. Это делается потому, что пользователи обычно намерены использовать дополнительные элементы, но не осознают, что они не установлены, или что установка Aer или IBM Quantum Provider завершилась ошибкой по какой-то причине. Если Вы хотите скрыть эти предупреждения, добавьте:

import warnings
warnings.filterwarnings('ignore', category=RuntimeWarning,
                        module='qiskit')

перед тем как импортировать компоненты qiskit в код. Это скроет предупреждение об отсутствии qiskit-aer и qiskit-ibmq-provider, а другие предупреждения из qiskit или других пакетов продолжат отображаться.

Установка Aer из исходного кода¶

  1. Склонируйте репозиторий Aer.

    git clone https://github.com/Qiskit/qiskit-aer

  2. Установите требования для сборки.

    pip install cmake scikit-build cython

После этого шаги по установке Aer зависят от используемой Вами операционной системы. Поскольку Aer является скомпилированной программой C++ с интерфейсом на Python, существуют другие зависимости помимо Python для построения бинарного файла Aer, который не может быть установлен без зависимости от операционной системы.

  1. Установите требуемые компилятору компоненты.

    Для сборки Aer требуется C++ компилятор и заголовки разработки (development headers).

    Если Вы используете Fedora или аналогичный дистрибутив Linux, используйте для установки:

    dnf install @development-tools

    Для Ubuntu/Debian, используйте:

    apt-get install build-essential

  2. Установите заголовки разработки (development headers) OpenBLAS.

    Если Вы используете Fedora или аналогичный дистрибутив Linux, используйте для установки:

    dnf install openblas-devel

    Для Ubuntu/Debian, используйте:

    apt-get install libopenblas-dev

  1. Установите зависимости.

    Чтобы использовать Clang компилятор в macOS, Вам нужно установить дополнительную библиотеку поддержки OpenMP. Вы можете использовать brew для установки этой и других зависимостей.

    brew install libomp

  2. Затем установите BLAS; по умолчанию используется OpenBLAS.

    brew install openblas

    Далее, установите Xcode Command Line Tools.

    xcode-select --install

В Windows необходимо использовать Anaconda3 <https://www.anaconda.com/distribution/#windows> __ или Miniconda3 <https://docs.conda.io/en/latest/miniconda.html> __ для установки всех зависимостей.

  1. Установите требуемые компилятору компоненты.

    conda install --update-deps vs2017_win-64 vs2017_win-32 msvc_runtime

  2. Установите бинарные зависимости и зависимости для сборки.

    conda install --update-deps -c conda-forge -y openblas cmake

  1. Постройте и установите qiskit-aer.

    Если у Вас установлен pip <19.0.0 и Ваша среда не требует пользовательской сборки, запустите:

    cd qiskit-aer
pip install .

    Это запустит сборку бинарных файлов и установит Aer.

    В качестве альтернативы, если у Вас установлен новый pip или у Вас есть особые требования, Вы можете создать Python wheel вручную.

    cd qiskit-aer
python ./setup.py bdist_wheel

    Если Вам нужно установить пользовательский параметр во время сборки, обратитесь к Пользовательские параметры во время сборки.

    Результат сборки будет храниться в каталоге dist/ репозитория Aer. Точная версия зависит от используемой

    cd dist
pip install qiskit_aer-*.whl

    Точное название файла зависит от текущей версии Aer.

Пользовательские параметры во время сборки¶

Система сборки Aer использует scikit-build для запуска компиляции при сборке с помощью Python интерфейса. Он выступает в качестве интерфейса setuptools для вызова CMake и компиляции бинарных файлов для Вашей системы.

Из-за сложности компиляции бинарных файлов Вам может потребоваться передать параметры определенной части процесса сборки. Способ их передачи:

python setup.py bdist_wheel [skbuild_opts] [-- [cmake_opts] [-- build_tool_opts]]

где элементы в квадратных скобках [] необязательны, и skbuild_opts, cmake_opts, build_tool_opts должны будут быть заменены на флаги на Ваш выбор. Список опций CMake доступен здесь: https://cmake.org/cmake/help/v3.6/manual/cmake.1.html#options. Например, Вы можете запустить следующее:

python setup.py bdist_wheel -- -- -j8

Это передает флаг -j8 базовой системе сборки (что в данном случае является Automake), говоря ей о том, что Вы хотите использовать 8 процессов для параллельной сборки.

Например, общий вариант использования этих флагов в Linux - указать конкретную версию компилятора C++ (если использующийся по умолчанию слишком старый):

python setup.py bdist_wheel -- -DCMAKE_CXX_COMPILER=g++-7

что даст инструкцию CMake использовать команду g++-7 вместо g++ по умолчанию при компиляции Aer.

Другим распространенным вариантом использования, в зависимости от среды - указать название Вашей платформы и отключить статическую связь (static linking).

python setup.py bdist_wheel --plat-name macosx-10.9-x86_64 \
-- -DSTATIC_LINKING=False -- -j8

Здесь -- plat-name - флаг setuptools для указания названия платформы для использования в метаданных пакета; -DSTATIC_LINKING - флаг CMake для отключения статического связывания, а -j8 - флаг Automake для использования 8 процессов для компиляции.

Установка Ignis из исходного кода¶

  1. Клонируйте репозиторий Ignis.

    git clone https://github.com/Qiskit/qiskit-ignis.git

  2. Клонирование репозитория создает локальную папку с названием qiskit-ignis.

    cd qiskit-ignis

  3. Если Вы хотите запустить тесты, установите необходимые инструменты для разработчика. Необязательно использовать пакет qiskit-ignis при установке из исходного кода.

    pip install -r requirements-dev.txt

  4. Установите Ignis.

    pip install .

Если Вы хотите установить пакет в редактируемом режиме, который позволяет не переустанавливать пакет в случае изменения кода в проекте, Вы можете сделать это с помощью:

pip install -e .

Qiskit Versioning¶

The Qiskit project is made up of several elements each performing different functionality. Each is independently useful and can be used on their own, but for convenience we provide this repository and meta-package to provide a single entrypoint to install all the elements at once. This is to simplify the install process and provide a unified interface to end users. However, because each Qiskit element has it’s own releases and versions some care is needed when dealing with versions between the different repositories. This document outlines the guidelines for dealing with versions and releases of both Qiskit elements and the meta-package.

For the rest of this guide the standard Semantic Versioning nomenclature will be used of: Major.Minor.Patch to refer to the different components of a version number. For example, if the version number was 0.7.1, then the major version is 0, the minor version 7, and the patch version 1.

Meta-package Version¶

The Qiskit meta-package version is an independent value that is determined by the releases of each of the elements being tracked. Each time we push a release to a tracked component (or add an element) the meta-package requirements, and version will need to be updated and a new release published. The timing should be coordinated with the release of elements to ensure that the meta-package releases track with element releases.

Adding New Tracked Elements¶

When a new Qiskit element is being added to the meta-package requirements, we need to increase the Minor version of the meta-package.

For example, if the meta-package is tracking 2 elements qiskit-aer and qiskit-terra and it’s version is 0.7.4. Then we release a new element qiskit-ignis that we intend to also have included in the meta-package. When we add the new element to the meta-package we increase the version to 0.8.0.

Patch Version Increases¶

When any Qiskit element that is being already tracked by the meta-package releases a patch version to fix bugs in a release we need also bump the requirement in the setup.py and then increase the patch version of the meta-package.

For example, if the meta-package is tracking 3 elements qiskit-terra==0.8.1, qiskit-aer==0.2.1, and qiskit-ignis==0.1.4 with the current version 0.9.6. When qiskit-terra release a new patch version to fix a bug 0.8.2 the meta-package will also need to increase it’s patch version and release, becoming 0.9.7.

Additionally, there are occasionally packaging or other bugs in the meta-package itself that need to be fixed by pushing new releases. When those are encountered we should increase the patch version to differentiate it from the broken release. Do not delete the broken or any old releases from pypi in any situation, instead just increase the patch version and upload a new release.

Minor Version Increases¶

Besides adding a new element to the meta-package the minor version of the meta-package should also be increased anytime a minor version is increased in a tracked element.

For example, if the meta-package is tracking 2 elements qiskit-terra==0.7.0 and qiskit-aer==0.1.1 and the current version is 0.7.5. When the qiskit-aer element releases 0.2.0 then we need to increase the meta-package version to be 0.8.0 to correspond to the new release.

Major Version Increases¶

The major version is different from the other version number components. Unlike the other version number components, which are updated in lock step with each tracked element, the major version is only increased when all tracked versions are bumped (at least before 1.0.0). Right now all the elements still have a major version number component of 0 and until each tracked element in the meta-repository is marked as stable by bumping the major version to be >=1 then the meta-package version should not increase the major version.

The behavior of the major version number component tracking after when all the elements are at >=1.0.0 has not been decided yet.

Optional Extras¶

In addition to the tracked elements there are additional packages built on top of Qiskit which are developed in tandem with Qiskit for example the application repositories like qiskit-optimization. For convienence these packages are tracked by the Qiskit metapackage as optional extras that can be installed with Qiskit. Releases of these optional downstream projects do not trigger a metapackage release as they are unpinned and do not effect the metapackage version. If there is a compatibility issue between Qiskit and these downstream optional dependencies and the minimum version needs to be adjusted in a standalone release this will only be done as a patch version release as it’s a packaging bugfix.

Qiskit Element Requirement Tracking¶

While not strictly related to the meta-package and Qiskit versioning how we track the element versions in the meta-package’s requirements list is important. Each element listed in the setup.py should be pinned to a single version. This means that each version of Qiskit should only install a single version for each tracked element. For example, the requirements list at any given point should look something like:

requirements = [
    "qiskit_terra==0.7.0",
    "qiskit-aer==0.1.1",
]

This is to aid in debugging, but also make tracking the versions across multiple elements more transparent.
