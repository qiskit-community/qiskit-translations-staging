Utiliser le Scheduler¶

L’ordonnanceur (Scheduler) va traduire un QuantumCircuit en une Pulse Schedule, en utilisant des calibrations des ´´gates´´ et des mesures: une description optimisée au niveau d’impulsion d’une opération sur des qubits particuliers.

Les systèmes d’exécution (Backends) qui sont activés avec OpenPulse auront généralement des calibrations définis pour des mesures et pour chacune de ses portes (gates) de base. Les calibrations peuvent également être définis ou mis à jour par l’utilisateur.

Utilisation de base¶

Pour commencer, construisez un circuit quantique comme vous le feriez normalement. Pour notre exemple ci-dessous, nous créons un simple état de Bell.

[1]:

from qiskit import QuantumCircuit

circ = QuantumCircuit(2, 2)
circ.h(0)
circ.cx(0, 1)
circ.measure([0, 1], [0, 1])

circ.draw()

[1]:

     ┌───┐     ┌─┐
q_0: ┤ H ├──■──┤M├───
     └───┘┌─┴─┐└╥┘┌─┐
q_1: ─────┤ X ├─╫─┤M├
          └───┘ ║ └╥┘
c: 2/═══════════╩══╩═
                0  1

Nous allons utiliser le backend simulé, FakeAlmaden, pour montrer comment utiliser l’ordonnanceur (Scheduler). Il contient des calibrations par défaut pour la mesure et pour ses portes (gates) de base. L’opération Hadamard n’est pas une de ces portes de base, donc nous utilisons le transpileur pour compiler notre circuit d’entrée pour générer un circuit équivalent en termes de portes de base de l’appareil.

[2]:

from qiskit import transpile, schedule as build_schedule
from qiskit.test.mock import FakeAlmaden

backend = FakeAlmaden()

transpiled_circ = transpile(circ, backend)  # Undefined Hadamard is replaced by U1
schedule = build_schedule(transpiled_circ, backend)

Voyons comment notre programme planning a été construit à partir du circuit transpiled_circ regarde. (Nous sélectionnons une plage pour le grahpique parce que la mesure et l’acquisition d’impulsions sont longues.)

[3]:

schedule.draw(plot_range=[0, 1700])

[3]:

Cela couvre les éléments de base ! Nous avons utilisé le transpileur pour réécrire le circuit en termes de portes (´´gates´´) de base, puis utilisé les calibrations par défaut de backend pour programmer le circuit transpilé. Ensuite, nous allons passer à l’ordonnancement avec des calibrations que nous construirons nous-mêmes.

Ordonnance avec des définitions de porte personnalisées¶

En utilisant le planificateur avec Pulse, nous ne sommes pas limités par les portes de base de l’appareil. Par exemple, nous pouvons définir un calibrage pour la porte d’Hadamard sur le qubit 0, qui est utilisé par notre circuit d’état Bell. Notre Hadamard sera une impulsion gaussienne. En pratique, la forme de l’impulsion et ses paramètres seraient optimisés grâce à une série d’expériences Rabi (voir le manuel  » Qiskit Textbook <https://qiskit.org/textbook/ch-quantum-hardware/calibrating-qubits-openpulse.html>` __ pour un aperçu).

[4]:

from qiskit.pulse import Schedule, Play, Gaussian, DriveChannel

h_q0 = Schedule(Play(Gaussian(duration=128, amp=0.045, sigma=8), DriveChannel(0)))

Nous ajoutons la calibration h_q0 à la InstructionScheduleMap existante du backend contenant la recherche de calibration d’instructions. Il doit être ajouté par le nom de la porte – ici nous utilisons le nom de la porte standard 'h' – et pour chauqe qubit. Rappelez-vous qu’une porte calibrée sur un qubit n’effectuera pas la même porte sur un autre qubit.

[5]:

cals = backend.defaults().instruction_schedule_map
cals.add('h', 0, h_q0)

schedule = build_schedule(circ, backend, inst_map=cals)

Remarquez comment cette fois-ci, nous avons pu ordonnancer circ, ce qui n’a pas été transpilé, donc il a toujours la porte d’Hadamard. La nouvelle calibration h_q0 a été utilisé pour ordonnancer circ.

Construction de circuits avec des portes non standard¶

Dans la section précédente, nous avons ajouté une définition native pour une porte (gate) standard. Dans cette section, nous construisons une porte entièrement nouvelle.

La première étape est de créer une définition pour cela en tant qu’Impulsion Schedule. Ci-dessous, nous démontrons également comment ces ordonnanes (Schdeules) peuvent être construits pour accepter des paramètres.

[6]:

import numpy as np

from qiskit.pulse import Play, Waveform, ControlChannel
from qiskit.pulse.schedule import ParameterizedSchedule

def random_pulse(duration):
    if duration < 1:
        raise ValueError("Duration cannot be less than 1.")
    return Waveform(np.random.random(duration))

def random_2q_gate(x):
    sched = Schedule()
    sched += Play(random_pulse(x), DriveChannel(0))
    sched += Play(random_pulse(x), ControlChannel(0))
    sched += Play(random_pulse(x), DriveChannel(1))
    return sched

my_random_2q_gate = ParameterizedSchedule(random_2q_gate, parameters=['x'])

Nous appelons notre opération d’ordonnance 'my_2q', et la définissons sur qubits 0 et 1 alors que nous l’ajoutons à nos calibrations.

[7]:

cals.add('my_2q', (0, 1), my_random_2q_gate)

Pour préparer notre circuit à utiliser cette définition, nous devons créer une nouvelle Gate avec le même nom que celui que nous avons défini ci-dessus, 'my_2q'.

[8]:

from qiskit.circuit import Gate

my_2q = Gate('my_2q', num_qubits=2, params=[100])  # Here we pass `duration`, as 100

Maintenant, nous pouvons construire notre circuit. Lorsque nous utilisons la nouvelle porte, ` ` my_2q ` , nous ` ` ajoutons ` ` (``append`) cela à notre circuit, et autrement construisons le circuit normalement.

[9]:

circ = QuantumCircuit(2, 2)
circ.h(0)
circ.append(my_2q, [0, 1])
circ.h(0)
circ.measure(0, 0)
circ.draw()

[9]:

q_0: ┤ H ├┤0            ├┤ H ├┤M├

Toutes les opérations dans circ sont définies dans nos cals, donc nous pourrions les planifier directement, mais nous allons supprimer notre calibration de Hadamard qubit 0 afin que nous puissions démontrer comment utiliser le transpileur avec notre nouvelle porte 2Q.

[10]:

cals.remove('h', 0)

transpiled_circ = transpile(circ,
                            coupling_map=[[0, 1]],
                            basis_gates=backend.configuration().basis_gates + ['my_2q'])
transpiled_circ.draw()

Maintenant, nous pouvons ordonnancer notre transpiled_circ comme nous l’avons fait auparavant.

[11]:

schedule = build_schedule(transpiled_circ, backend, inst_map=cals)

schedule.draw(plot_range=(0, 500))

Méthode d’ordonnance (scheduler)¶

L’ordonnanceur peut suivre plusieurs routines. Par défaut, il suit une règle * aussi tard que possible * (ALAP, as late as possible). Une autre méthode d’ordonnance est * dès que possible * (ASAP, as soon as possible). Pour les deux méthodes, le planning de sortie est minimal: dans le chemin d’opération de durée la plus longue du circuit d’entrée, le moment de début de chaque opération correspond au moment de fin de l’opération procédédente. Les méthodes déterminent comment planifier (schedule) des opérations en dehors de ce chemin le plus long.

Ceci devient clair à travers un exemple:

[12]:

circ = QuantumCircuit(2, 2)
circ.x(0)
circ.x(0)
circ.x(1)
circ.measure([0, 1], [0, 1])

[12]:

<qiskit.circuit.instructionset.InstructionSet at 0x138758470>

Pour notre backend simulé, ces opérations de :math:` X sont toutes de la même durée (` ` durée = 160 dt ` ). Pour que le planning (``schedule`) soit minimal, les deux opérations :math:` X’sur le qubit 0 seront planifiées consécutives (back-to-back), et l’impulsion de mesure suivra immédiatement la seconde de ces impulsions.

ALAP choisira le dernier moment valide pour ordonnancer des opérations moins restreintes, de sorte que la porte :math:` X’sur le qubit 1 exécutera en synchronisation avec la deuxième porte :math:` X’sur le qubit 0.

[13]:

schedule = build_schedule(circ, backend,
                          method="as_late_as_possible")
schedule.draw(channels=[DriveChannel(0), DriveChannel(1)], scale=5)

[13]:

D’autre part, comme son nom l’indique, ASAP ordonnancera des opérations dès que ses ressources seront libres. Ainsi, la porte :math:` X’sur le qubit 1 sera programmée à ` ` time=0 ` , en synchronisation avec la première porte :math: X’sur le qubit 0.

[14]:

schedule = build_schedule(circ, backend,
                          method="as_soon_as_possible")
schedule.draw(channels=[DriveChannel(0), DriveChannel(1)], scale=5)

[14]:

ALAP est la valeur par défaut car elle permet aux qubits de rester inactifs aussi longtemps que possible. Dans ce cas, la différence entre ALAP et ASAP peut être négligeable, mais dans ALAP, le qubit 0 n’a pas le temps de décomposition (decay) à partir de l’état excité avant la mesure.
