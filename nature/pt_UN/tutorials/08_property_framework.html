O Framework Property¶

[1]:

import numpy as np

O Qiskit Nature 0.2.0 apresenta o framework Property. Esse framework substitui os tipos de retorno de driver legados como o QMolecule e WatsonHamiltonian com uma abordagem mais modular e extensível.

Neste tutorial, vamos percorrer o framework, explicaremos seus componentes mais importantes e mostraremos como você poderá amplia-lo com uma property personalizada.

O que é uma Property?¶

Na sua essência, uma Property é um objeto que complementa alguns dados brutos com funções que lhe permitirão transformar/manipular/interpretar estes dados brutos. Esta «definição» se mantém bastante abstrata a propósito, mas o que isso significa é essencialmente o seguinte. Uma Property:

  • representa um observável físico (estes são os dados brutos)

  • pode ser expressado como um operador

  • pode ser avaliado com uma função de onda

  • proporciona um método interpret que da significado ao autovalor do operador qubit avaliado

[2]:

from qiskit_nature.properties import Property, GroupedProperty

O módulo qiskit_nature.properties fornece duas classes:

  1. Property: esta é a interface básica. Requer apenas um name e um método ìnterpret` para ser implementada.

  2. GroupedProperty: esta classe é uma implementação de ``Composite pattern <https://en.wikipedia.org/wiki/Composite_pattern>`__ que lhe permite agrupar múltiplas propriedades em uma.

Nota: Propriedades agrupadas devem possuir atributos name únicos!

Propriedades de Segunda Quantização¶

No momento da escrita, Qiskit Nature envia com uma única variante de propriedades: os objetos SecondQuantizedProperty.

Este subtipo agrega um requisito adicional por qualquer SecondQuantizedProperty

  • deve implementar um método second_q_ops``que constrói uma (lista de) ``SecondQuantizedOps.

O módulo qiskit_nature.properties.second_quantization é dividido entre os módulos electronic e vibrational (similares aos outros módulos de Qiskit Nature). Vamos mergulhar no sub-módulo electronic primeiro.

ElectronicEnergy é sem dúvida a propriedade mais importante porque contém Hamiltoniano que representa o problema de estrutura eletrônica cujos autovalores nos interessam ao resolver um ElectronicStructureProblem. A maneira como ela armazena o Hamiltoniano é, assim como o restante do framework, altamente modular. O inicializador da ElectronicEnergy tem um único argumento obrigatório, electronic_integrals, que é uma `` List[ElectronicIntegrals].

ElectronicIntegrals¶

[4]:

Os ElectronicIntegrals são uma classe contêiner para interações de n-corpos em uma determinada base, que pode ser qualquer uma das seguintes:

Tomemos o OneBodyElectronicIntegrals como exemplo. Como o nome sugere, este contêiner é para integrais de interação de 1-corpo. Você pode construir uma instância dele da seguinte forma:

[7]:

one_body_ints = OneBodyElectronicIntegrals(
    ElectronicBasis.MO,
    (
        np.eye(2),
        2 * np.eye(2),
    ),
)
print(one_body_ints)

Como pode ver, o primeiro argumento simplesmente especifica a base das integrais. O segundo argumento requer um pouco mais de explicação:

  1. No caso das bases AO ou MO, este argumento deve ser um par de matrizes numpy, onde a primeira e a segunda são as integrais de spin alfa e beta, respectivamente.

Nota: O segundo argumento pode ser None, para o caso onde as integrais de spin beta são as iguais às integrais de spin alfa (logo, não há necessidade de fornecer os mesmos valores duas vezes).

  1. No caso da base SO, este argumento deve ser uma matriz numpy, armazenando as integrais alpha- e beta-spin em ordem bloqueada, como por exemplo:

spin_basis = np.block([[alpha_spin, zeros], [zeros, beta_spin]])

As TwoBodyElectronicIntegrals funcionam basicamente da mesma forma, exceto que contém todas as possíveis spin-combinations na tupla de matrizes numpy. Por exemplo:

  • as matrizes numpy devem ser ordenadas como ("alpha-alpha", "beta-alpha", "beta-beta", "alpha-beta")

Interpretação do resultado¶

Um benefício adicional que obtemos da estrutura Property, é que a interpretação do resultado de um autovalor calculado pode ser manipulado por cada propriedade em si. Isto resulta em classes agradáveis e logicamente consistentes pois estes resultados são interpretados no mesmo contexto em que os dados brutos são utilizados.

[11]:

from qiskit_nature.results import ElectronicStructureResult

# some dummy result
result = ElectronicStructureResult()
result.eigenenergies = np.asarray([-1])
result.computed_energies = np.asarray([-1])


# now, let's interpret it
electronic_energy.interpret(result)
print(result)

=== GROUND STATE ENERGY ===

* Electronic ground state energy (Hartree): -1
  - computed part:      -1

Embora este exemplo em particular ainda não seja muito impressionante, aguarde até utilizarmos mais propriedades de uma vez.

ParticleNumber¶

A propriedade ParticleNumber também ocupa um lugar especial entre as propriedades integradas porque serve a um duplo propósito:

  • armazenando o número de partículas no sistema eletrônico

  • fornecer os operadores que avaliam o número de partículas para determinar uma solução automática do problema

Portanto, esta propriedade é necessária caso deseje utilizar funcionalidades adicionais como ActiveSpaceTransformer ou ElectronicStructureProblem.default_filter_criterion().

[12]:

particle_number = ParticleNumber(
    num_spin_orbitals = 4,
    num_particles = (1, 1),
)
print(particle_number)

Escrevendo propriedades personalizadas¶

Você pode estender a estrutura da Property com suas próprias implementações. Aqui, vamos percorrer o exemplo simples de construção de uma propriedade para a * densidade eletrônica *. Uma vez que este observável é essencialmente baseado em matrizes, iremos alavancar o contêiner OneBodyElectronicIntegrals para armazenar a matriz de densidade real.

class ElectronicDensity(ElectronicProperty):
    """A simple electronic density property.

    This basic example works only in the MO basis!
    """

    def __init__(self, num_molecular_orbitals: int) -> None:
        super().__init__(self.__class__.__name__)
        self._num_molecular_orbitals = num_molecular_orbitals

    def __str__(self) -> str:
        string = [super().__str__() + ":"]
        string += [f"\t{self._num_molecular_orbitals} MOs"]
        return "\n".join(string)

    @classmethod
    def from_legacy_driver_result(cls, result) -> "ElectronicDensity":
        cls._validate_input_type(result, QMolecule)

        qmol = cast(QMolecule, result)

        return cls(qmol.num_molecular_orbitals)


    def second_q_ops(self) -> List[FermionicOp]:
        ops = []

        # iterate all pairs of molecular orbitals
        for mo_i, mo_j in product(range(self._num_molecular_orbitals), repeat=2):

            # construct an auxiliary matrix where the only non-zero entry is at the current pair of MOs
            number_op_matrix = np.zeros((self._num_molecular_orbitals, self._num_molecular_orbitals))
            number_op_matrix[mo_i, mo_j] = 1

            # leverage the OneBodyElectronicIntegrals to construct the corresponding FermionicOp
            one_body_ints = OneBodyElectronicIntegrals(ElectronicBasis.MO, (number_op_matrix, number_op_matrix))
            ops.append(one_body_ints.to_second_q_op())

        return ops

    def interpret(self, result) -> None:
        # here goes the code which interprets the eigenvalues returned for the auxiliary operators
        pass

Claro, o exemplo acima é mínimo e pode ser extendido à vontade.

Nota: a partir do Qiskit Nature versão 0.2.0, a integração direta de objetos de propriedade personalizados na pilha ainda não foi implementada, devido às limitações da análise do operador auxiliar. Consulte https://github.com/Qiskit/qiskit-nature/issues/312 para obter mais detalhes.

Por enquanto, você ainda pode avaliar uma propriedade personalizada, passando seus operadores gerados diretamente para o método Eigensolver.solve por meio da construção do aux_operators. Observe, no entanto, que você terá que lidar com as transformações aplicadas às suas propriedades manualmente, até que o problema acima seja resolvido.
