El Framework Property¶

[1]:

import numpy as np

Qiskit Nature 0.2.0 presenta el framework Property. Este framework reemplaza los tipos de retorno de controladores heredados como QMolecule y WatsonHamiltonian con un enfoque más modular y extensible.

En este tutorial, recorreremos el framework, explicaremos sus componentes más importantes y te mostraremos cómo puedes ampliarlo con una property personalizada.

¿Qué es una Property?¶

En esencia, una Property (propiedad) es un objeto que complementa algunos datos sin procesar con funciones que te permiten transformar/manipular/interpretar estos datos sin procesar. Esta «definición» se mantiene bastante abstracta a propósito, pero lo que significa es esencialmente lo siguiente. Una Property:

  • representa un observable físico (esos son los datos sin procesar)

  • se puede expresar como un operador

  • se puede evaluar con una función de onda

  • proporciona un método interpret que da significado al valor propio del operador de qubit evaluado

[2]:

from qiskit_nature.properties import Property, GroupedProperty

El módulo qiskit_nature.properties proporciona dos clases:

  1. Property: esta es la interfaz básica. Requiere solo un name y un método interpret para ser implementada.

  2. GroupedProperty: esta clase es una implementación del Patrón de diseño Composite que te permite agrupar múltiples propiedades en una.

Nota: ¡Las propiedades agrupadas deben tener atributos únicos de name!

Propiedades de la Segunda Cuantización¶

En el momento de escribir este artículo, Qiskit Nature se envía con una única variante de propiedades: los objetos SecondQuantizedProperty.

Este subtipo agrega un requisito adicional porque cualquier SecondQuantizedProperty

  • debe implementar un método second_q_ops que construya una (lista de) SecondQuantizedOps.

El módulo qiskit_nature.properties.second_quantization se divide a su vez en los módulos electronic y vibrational (similares a otros módulos de Qiskit Nature). Vamos a sumergirnos primero en el submódulo electronic.

Propiedades de la Segunda Cuantización Electrónica¶

Qiskit Nature, listo para usar, envía las siguientes propiedades electrónicas:

ElectronicEnergy es posiblemente la propiedad más importante porque contiene al Hamiltoniano que representa el problema de la estructura electrónica cuyos valores propios nos interesan al resolver un ElectronicStructureProblem. La forma en que almacena a este Hamiltoniano es, al igual que el resto del framework, muy modular. El inicializador de ElectronicEnergy tiene un único argumento requerido, electronic_integrals, que es una List[ElectronicIntegrals].

ElectronicIntegrals¶

[4]:

from qiskit_nature.properties.second_quantization.electronic.integrals import (
    ElectronicIntegrals,
    OneBodyElectronicIntegrals,
    TwoBodyElectronicIntegrals,

Las ElectronicIntegrals son una clase de contenedor para interacciones de n-cuerpos en una base determinada, que puede ser cualquiera de las siguientes:

[5]:

Tomemos como ejemplo los OneBodyElectronicIntegrals. Como sugiere el nombre, este contenedor es para integrales de interacción de 1 cuerpo. Puedes construir una instancia de la siguiente manera:

[7]:

one_body_ints = OneBodyElectronicIntegrals(
    ElectronicBasis.MO,
    (
        np.eye(2),
        2 * np.eye(2),
    ),
)
print(one_body_ints)

Como puedes ver, el primer argumento simplemente especifica la base de las integrales. El segundo argumento requiere una pequeña explicación más:

  1. En el caso de las bases AO o MO, este argumento debe ser un par de matrices numéricas, donde la primera y la segunda son las integrales de espín alfa y beta, respectivamente.

Nota: El segundo argumento puede ser None, para el caso donde las integrales beta-spin son las mismas que las integrales alfa-spin (por lo que no es necesario proporcionar los mismos valores dos veces).

  1. En el caso de la base SO, este argumento debe ser una matriz numérica única, almacenando las integrales de espín alfa y beta en orden bloqueado, es decir, así:

spin_basis = np.block([[alpha_spin, zeros], [zeros, beta_spin]])

Debemos tomar nota de algunas observaciones:

  • las matrices numpy se ordenarán como ("alpha-alpha", "beta-alpha", "beta-beta", "alpha-beta")

  • la matriz alpha-alpha no puede ser None

  • si la matriz alpha-beta es None, pero beta-alpha no lo es, se usará su transposición (como arriba)

  • en cualquier otro caso, las matrices que son None se llenarán con la matriz alpha-alpha

  • en el caso de la base SO, se debe especificar una única matriz numérica. Refiérete a TwoBodyElectronicIntegrals.to_spin() para conocer su formato exacto.

Interpretación del resultado¶

Un beneficio adicional que obtenemos del framework Property es que la interpretación del resultado de un valor propio calculado puede ser manejada por cada propiedad en sí. Esto da como resultado clases agradables y lógicamente consistentes porque el resultado se interpreta en el mismo contexto donde están disponibles los datos sin procesar.

[11]:

from qiskit_nature.results import ElectronicStructureResult

# some dummy result
result = ElectronicStructureResult()
result.eigenenergies = np.asarray([-1])
result.computed_energies = np.asarray([-1])


# now, let's interpret it
electronic_energy.interpret(result)
print(result)

=== GROUND STATE ENERGY ===

* Electronic ground state energy (Hartree): -1
  - computed part:      -1

Si bien este ejemplo en particular aún no es muy impresionante, espera hasta que usemos más propiedades a la vez.

ParticleNumber¶

La propiedad ParticleNumber también ocupa un lugar especial entre las propiedades integradas porque tiene un doble propósito:

  • almacenar el número de partículas en el sistema electrónico

  • proveer los operadores que evalúan el número de partículas para una determinada resolución del problema

Por lo tanto, esta propiedad es necesaria si desea utilizar funciones adicionales como ActiveSpaceTransformer o el ElectronicStructureProblem.default_filter_criterion().

[12]:

particle_number = ParticleNumber(
    num_spin_orbitals = 4,
    num_particles = (1, 1),
)
print(particle_number)

Escribir Propiedades personalizadas¶

Puedes ampliar el framework Property con tus propias implementaciones. Aquí, veremos el ejemplo simple de construir una propiedad para la densidad electrónica. Dado que este observable se basa esencialmente en matrices, aprovecharemos el contenedor OneBodyElectronicIntegrals para almacenar la matriz de densidad real.

from typing import List

from qiskit_nature.properties.second_quantization.electronic.integrals import OneBodyElectronicIntegrals

class ElectronicDensity(ElectronicProperty):
    """A simple electronic density property.

    This basic example works only in the MO basis!
    """

    def __init__(self, num_molecular_orbitals: int) -> None:
        super().__init__(self.__class__.__name__)
        self._num_molecular_orbitals = num_molecular_orbitals

    def __str__(self) -> str:
        string = [super().__str__() + ":"]
        string += [f"\t{self._num_molecular_orbitals} MOs"]
        return "\n".join(string)

    @classmethod
    def from_legacy_driver_result(cls, result) -> "ElectronicDensity":
        cls._validate_input_type(result, QMolecule)

        qmol = cast(QMolecule, result)

        return cls(qmol.num_molecular_orbitals)


    def second_q_ops(self) -> List[FermionicOp]:
        ops = []

        # iterate all pairs of molecular orbitals
        for mo_i, mo_j in product(range(self._num_molecular_orbitals), repeat=2):

            # construct an auxiliary matrix where the only non-zero entry is at the current pair of MOs
            number_op_matrix = np.zeros((self._num_molecular_orbitals, self._num_molecular_orbitals))
            number_op_matrix[mo_i, mo_j] = 1

            # leverage the OneBodyElectronicIntegrals to construct the corresponding FermionicOp
            one_body_ints = OneBodyElectronicIntegrals(ElectronicBasis.MO, (number_op_matrix, number_op_matrix))
            ops.append(one_body_ints.to_second_q_op())

        return ops

    def interpret(self, result) -> None:
        # here goes the code which interprets the eigenvalues returned for the auxiliary operators
        pass

Por supuesto, el ejemplo anterior es mínimo y puede ampliarse a voluntad.

Nota: a partir de la versión 0.2.0 de Qiskit Nature, la integración directa de los objetos Property personalizados en la pila aún no está implementada, debido a las limitaciones del análisis del operador auxiliar. Consulta https://github.com/Qiskit/qiskit-nature/issues/312 para obtener más detalles.

Por el momento, aún puedes evaluar una Property personalizada, pasando los operadores generados directamente al método Eigensolver.solve mediante la construcción de los aux_operators. Sin embargo, ten en cuenta que tendrás que lidiar con las transformaciones aplicadas a tus propiedades manualmente, hasta que se resuelva el problema anterior.
