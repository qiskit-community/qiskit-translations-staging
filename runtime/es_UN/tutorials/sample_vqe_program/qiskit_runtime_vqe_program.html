Creación de Programas Personalizados para Qiskit Runtime¶

Paul Nation

Equipo de Habilitación Técnica de IBM Quantum Partners

Aquí demostraremos cómo crear, cargar y usar un programa personalizado para Qiskit Runtime. Como la utilidad del motor de ejecución Runtime radica en su capacidad para ejecutar muchos circuitos cuánticos con latencias bajas, este tutorial te mostrará cómo crear tu propio programa Solucionador Propio Variacional Cuántico (Variational Quantum Eigensolver, VQE) desde cero.

Requisitos previos¶

  • Debes tener instalado el último Qiskit.

  • Debes tener una cuenta de IBM Cloud o IBM Quantum que pueda acceder a Qiskit Runtime.

Limitaciones actuales¶

El motor de ejecución de runtime actualmente tiene las siguientes limitaciones que deben tenerse en cuenta:

  • Las imágenes de Docker utilizadas por el runtime incluyen solo Qiskit y sus dependencias, con pocas excepciones. Una excepción es la inclusión del paquete de mitigación de medición mthree.

  • Por motivos de seguridad, el runtime no puede realizar llamadas de Internet fuera del entorno.

A medida que madure Qiskit Runtime, se eliminarán estas limitaciones.

VQE simple¶

VQE es un procedimiento híbrido cuántico-clásico de optimización que encuentra el estado propio y la energía propia más bajos de un sistema lineal definido por un operador Hamiltoniano de Pauli dado. Por ejemplo, considera el siguiente Hamiltoniano de dos qubits:

\[H = A X_{1}\otimes X_{0} + A Y_{1}\otimes Y_{0} + A Z_{1}\otimes Z_{0},\]

donde \(A\) es el coeficiente numérico y los subíndices etiquetan los qubits sobre los que actúan los operadores. El índice cero que está más a la derecha es el orden utilizado en Qiskit. Los operadores de Pauli nos dicen qué base de medición utilizar a la hora de medir cada uno de los qubits.

Queremos encontrar el estado fundamental (estado de energía más bajo) de este Hamiltoniano y el vector propio asociado. Para hacer esto, debemos comenzar en un estado inicial dado y variar iterativamente los parámetros que definen este estado utilizando un optimizador clásico, de modo que las energías calculadas de los pasos posteriores sean nominalmente más bajas que las anteriores. El estado parametrizado del sistema está definido por un circuito cuántico ansatz que debe tener un soporte distinto de cero en la dirección del estado fundamental. Debido a que, en general, no conocemos la solución, la elección del circuito ansatz puede ser muy específica del problema con una forma dictada por información adicional. Para obtener más información sobre los algoritmos variacionales, remitimos al lector a Nature Reviews Physics volume 3, 625 (2021).

Por lo tanto, necesitamos al menos las siguientes entradas para crear nuestro programa cuántico VQE:

  1. Una representación del Hamiltoniano que especifica el problema.

  2. Una opción de circuito ansatz parametrizado y la capacidad de pasar opciones de configuración, si las hay.

Sin embargo, las siguientes también son entradas beneficiosas que los usuarios podrían querer tener:

  1. Agregar la capacidad de pasar un estado inicial.

  2. Variar el número de iteraciones que son tomadas.

  3. Capacidad para seleccionar qué optimizador clásico se usa y establecer valores de configuración, si hay alguna.

  4. Capacidad para activar y desactivar la mitigación de medición.

Especificación de la forma de los valores de entrada¶

Todas las entradas a los programas runtime deben ser objetos serializables. Es decir, todo lo que pases a un programa runtime debe poder convertirse al formato JSON. Por lo tanto, es beneficioso mantener las entradas limitadas a estructuras y tipos de datos básicos, a menos que tengas experiencia con la serialización de objetos personalizados, o que sean tipos comunes de Qiskit, como QuantumCircuit, etc., que el RuntimeEncoder integrado pueda manejar. Afortunadamente, el programa VQE descrito anteriormente se puede hacer con componentes simples de Python.

Primero, es posible representar cualquier Hamiltoniano utilizando una lista de valores, cada uno de los cuales contiene el coeficiente numérico para cada término y la representación de cadena para los operadores de Pauli. Para el ejemplo anterior, la energía del estado fundamental con \(A=1\) es \(-3\) y podemos escribirla como:

[1]:

H = [(1, "XX"), (1, "YY"), (1, "ZZ")]

A continuación, debemos proporcionar la capacidad de especificar el circuito Ansatz parametrizado. Aquí aprovecharemos el hecho de que muchos circuitos ansatz están predefinidos en la Biblioteca de Circuitos de Qiskit. Se pueden encontrar ejemplos en la sección de circuitos N-local.

Nos gustaría que el usuario pudiera seleccionar entre las opciones de ansatz como: NLocal, TwoLocal, y EfficientSU2. Podríamos hacer que el usuario pase todo el circuito ansatz al programa; sin embargo, para reducir el tamaño de la carga, pasaremos el ansatz por nombre. En el programa runtime, podemos tomar este nombre y obtener la clase a la que corresponde de la biblioteca usando, por ejemplo,

[2]:

import qiskit.circuit.library.n_local as lib_local

ansatz = getattr(lib_local, "EfficientSU2")

Para la configuración de ansatz, pasaremos un simple dict de valores.

Opcionales¶

  • Si queremos agregar la capacidad de pasar un estado inicial, necesitaremos agregar la capacidad de pasar una lista 1D / arreglo NumPy. Debido a que la cantidad de parámetros depende del ansatz y su configuración, el usuario debería saber qué ansatz está usando con anticipación.

  • Seleccionar un número de iteraciones requiere simplemente pasar un valor entero.

  • Aquí permitiremos seleccionar un optimizador clásico por su nombre entre SciPy, y un dict de parámetros de configuración. Ten en cuenta que para la ejecución en un sistema real, el ruido inherente a los sistemas cuánticos actuales hace que tener un optimizador estocástico sea crucial para tener éxito. SciPy no tiene esa opción, y el integrado en Qiskit está envuelto de tal manera que dificulta su uso en otros lugares. Como tal, aquí usaremos un optimizador SPSA escrito para que coincida con el estilo de los de SciPy. Esta función se proporciona en el Apéndice A.

  • Finalmente, para la mitigación de errores de medición, simplemente podemos pasar un valor booleano (True/False).

Programa principal¶

Ahora estamos en condiciones de comenzar a construir nuestro programa principal. Sin embargo, antes de hacerlo, señalamos que hace que el código sea más limpio al crear una función separada que toma cadenas de operadores de Pauli que definen nuestro Hamiltoniano y las convierte en una lista de circuitos con compuertas de un solo qubit que cambian la base de medición para cada qubit, si es necesario. Esta función se proporciona en el Apéndice B.

Firma requerida¶

Cada programa runtime se define a través de la función main y debe tener la siguiente firma de entrada:

main(backend, user_message, *args, **kwargs)

donde backend es el backend en el que se ejecutará el programa, y user_message es la clase mediante la cual los resultados provisionales (y posiblemente finales) se comunican al usuario. Después de estos dos elementos, agregamos nuestros argumentos específicos del programa y argumentos de palabras clave.

El programa VQE principal¶

Aquí está el programa principal para nuestro VQE de muestra. Lo que hace cada elemento de la función está escrito en los comentarios antes de que aparezca el elemento.

Este programa utiliza el paquete de mitigación de mediciones mthree. Puedes instalar el paquete a través de pip install mthree.

[3]:

# Grab functions and modules from dependencies
import numpy as np
import scipy.optimize as opt
from scipy.optimize import OptimizeResult
import mthree

# Grab functions and modules from Qiskit needed
from qiskit import QuantumCircuit, transpile
import qiskit.circuit.library.n_local as lib_local

# The entrypoint for our Runtime Program
def main(
    backend,
    user_messenger,
    hamiltonian,
    ansatz="EfficientSU2",
    ansatz_config={},
    x0=None,
    optimizer="SPSA",
    optimizer_config={"maxiter": 100},
    shots=8192,
    use_measurement_mitigation=False,
):

    """
    The main sample VQE program.

    Parameters:
        backend (ProgramBackend): Qiskit backend instance.
        user_messenger (UserMessenger): Used to communicate with the
                                        program user.
        hamiltonian (list): Hamiltonian whose ground state we want to find.
        ansatz (str): Optional, name of ansatz quantum circuit to use,
                      default='EfficientSU2'
        ansatz_config (dict): Optional, configuration parameters for the
                              ansatz circuit.
        x0 (array_like): Optional, initial vector of parameters.
        optimizer (str): Optional, string specifying classical optimizer,
                         default='SPSA'.
        optimizer_config (dict): Optional, configuration parameters for the
                                 optimizer.
        shots (int): Optional, number of shots to take per circuit.
        use_measurement_mitigation (bool): Optional, use measurement mitigation,
                                           default=False.

    Returns:
        OptimizeResult: The result in SciPy optimization format.
    """

    # Split the Hamiltonian into two arrays, one for coefficients, the other for
    # operator strings
    coeffs = np.array([item[0] for item in hamiltonian], dtype=complex)
    op_strings = [item[1] for item in hamiltonian]
    # The number of qubits needed is given by the number of elements in the strings
    # the defiune the Hamiltonian. Here we grab this data from the first element.
    num_qubits = len(op_strings[0])

    # We grab the requested ansatz circuit class from the Qiskit circuit library
    # n_local module and configure it using the number of qubits and options
    # passed in the ansatz_config.
    ansatz_instance = getattr(lib_local, ansatz)
    ansatz_circuit = ansatz_instance(num_qubits, **ansatz_config)

    # Here we use our convenence function from Appendix B to get measurement circuits
    # with the correct single-qubit rotation gates.
    meas_circs = opstr_to_meas_circ(op_strings)

    # When computing the expectation value for the energy, we need to know if we
    # evaluate a Z measurement or and identity measurement.  Here we take and X and Y
    # operator in the strings and convert it to a Z since we added the rotations
    # with the meas_circs.
    meas_strings = [string.replace("X", "Z").replace("Y", "Z") for string in op_strings]

    # Take the ansatz circuits, add the single-qubit measurement basis rotations from
    # meas_circs, and finally append the measurements themselves.
    full_circs = [
        ansatz_circuit.compose(mcirc).measure_all(inplace=False) for mcirc in meas_circs
    ]

    # Get the number of parameters in the ansatz circuit.
    num_params = ansatz_circuit.num_parameters

    # Use a given initial state, if any, or do random initial state.
    if x0:
        x0 = np.asarray(x0, dtype=float)
        if x0.shape[0] != num_params:
            raise ValueError(
                "Number of params in x0 ({}) does not match number \
                              of ansatz parameters ({})".format(
                    x0.shape[0], num_params
                )
            )
    else:
        x0 = 2 * np.pi * np.random.rand(num_params)

    # Because we are in general targeting a real quantum system, our circuits must be transpiled
    # to match the system topology and, hopefully, optimize them.
    # Here we will set the transpiler to the most optimal settings where 'sabre' layout and
    # routing are used, along with full O3 optimization.

    # This works around a bug in Qiskit where Sabre routing fails for simulators (Issue #7098)
    trans_dict = {}
    if not backend.configuration().simulator:
        trans_dict = {"layout_method": "sabre", "routing_method": "sabre"}
    trans_circs = transpile(full_circs, backend, optimization_level=3, **trans_dict)

    # If using measurement mitigation we need to find out which physical qubits our transpiled
    # circuits actually measure, construct a mitigation object targeting our backend, and
    # finally calibrate our mitgation by running calibration circuits on the backend.
    if use_measurement_mitigation:
        maps = mthree.utils.final_measurement_mapping(trans_circs)
        mit = mthree.M3Mitigation(backend)
        mit.cals_from_system(maps)

    # Here we define a callback function that will stream the optimizer parameter vector
    # back to the user after each iteration.  This uses the `user_messenger` object.
    # Here we convert to a list so that the return is user readable locally, but
    # this is not required.
    def callback(xk):
        user_messenger.publish(list(xk))

    # This is the primary VQE function executed by the optimizer. This function takes the
    # parameter vector as input and returns the energy evaluated using an ansatz circuit
    # bound with those parameters.
    def vqe_func(params):
        # Attach (bind) parameters in params vector to the transpiled circuits.
        bound_circs = [circ.bind_parameters(params) for circ in trans_circs]

        # Submit the job and get the resultant counts back
        counts = backend.run(bound_circs, shots=shots).result().get_counts()

        # If using measurement mitigation apply the correction and
        # compute expectation values from the resultant quasiprobabilities
        # using the measurement strings.
        if use_measurement_mitigation:
            quasi_collection = mit.apply_correction(counts, maps)
            expvals = quasi_collection.expval(meas_strings)
        # If not doing any mitigation just compute expectation values
        # from the raw counts using the measurement strings.
        # Since Qiskit does not have such functionality we use the convenence
        # function from the mthree mitigation module.
        else:
            expvals = mthree.utils.expval(counts, meas_strings)

        # The energy is computed by simply taking the product of the coefficients
        # and the computed expectation values and summing them. Here we also
        # take just the real part as the coefficients can possibly be complex,
        # but the energy (eigenvalue) of a Hamiltonian is always real.
        energy = np.sum(coeffs * expvals).real
        return energy

    # Here is where we actually perform the computation.  We begin by seeing what
    # optimization routine the user has requested, eg. SPSA verses SciPy ones,
    # and dispatch to the correct optimizer.  The selected optimizer starts at
    # x0 and calls 'vqe_func' everytime the optimizer needs to evaluate the cost
    # function.  The result is returned as a SciPy OptimizerResult object.
    # Additionally, after every iteration, we use the 'callback' function to
    # publish the interim results back to the user. This is important to do
    # so that if the Program terminates unexpectedly, the user can start where they
    # left off.

    # Since SPSA is not in SciPy need if statement
    if optimizer == "SPSA":
        res = fmin_spsa(vqe_func, x0, args=(), **optimizer_config, callback=callback)
    # All other SciPy optimizers here
    else:
        res = opt.minimize(
            vqe_func, x0, method=optimizer, options=optimizer_config, callback=callback
        )
    # Return result. OptimizeResult is a subclass of dict.
    return res

Metadatos del programa¶

Los metadatos del programa son esencialmente la docstring para un programa runtime. Describe la información general del programa, como el name, description, y el max_execution_time en el que el programa se puede ejecutar, así como los detalles de las entradas y las salidas que espera el programa. Como mínimo, se requieren los valores descritos anteriormente

[9]:

meta = {
    "name": "sample-vqe",
    "description": "A sample VQE program.",
    "max_execution_time": 100000,
    "spec": {},
}

Es importante configurar el max_execution_time lo suficientemente alto para que tu programa no termine inesperadamente. Además, uno debe asegurarse de que los resultados intermedios se envíen de vuelta al usuario para que, si sucede algo, el usuario pueda comenzar donde lo dejó.

Sin embargo, es una buena forma detallar los parámetros y los tipos de devolución, así como los resultados intermedios. Dicho esto, si se crea un runtime destinado a ser utilizado por otros, es probable que esta información también se refleje en la firma de una función o clase con la que el usuario interactuaría directamente; los usuarios finales no deben llamar directamente a los programas runtime. Veremos por qué a continuación. Sin embargo, agreguemos a nuestros metadatos. Primero, la sección parameters detalla las entradas que el usuario puede pasar:

[10]:

meta["spec"]["parameters"] = {
    "$schema": "https://json-schema.org/draft/2019-09/schema",
    "properties": {
        "hamiltonian": {
            "description": "Hamiltonian whose ground state we want to find.",
            "type": "array",
        },
        "ansatz": {
            "description": "Name of ansatz quantum circuit to use, default='EfficientSU2'",
            "type": "string",
            "default": "EfficientSU2",
        },
        "ansatz_config": {
            "description": "Configuration parameters for the ansatz circuit.",
            "type": "object",
        },
        "optimizer": {
            "description": "Classical optimizer to use, default='SPSA'.",
            "type": "string",
            "default": "SPSA",
        },
        "x0": {
            "description": "Initial vector of parameters. This is a numpy array.",
            "type": "array",
        },
        "optimizer_config": {
            "description": "Configuration parameters for the optimizer.",
            "type": "object",
        },
        "shots": {
            "description": "The number of shots used for each circuit evaluation.",
            "type": "integer",
        },
        "use_measurement_mitigation": {
            "description": "Use measurement mitigation, default=False.",
            "type": "boolean",
            "default": False,
        },
    },
    "required": ["hamiltonian"],
}

A continuación, la sección return_values informa sobre los tipos de devolución:

[11]:

meta["spec"]["return_values"] = {
    "$schema": "https://json-schema.org/draft/2019-09/schema",
    "description": "Final result in SciPy optimizer format",
    "type": "object",
}

y finalmente, especifiquemos qué regresa cuando se devuelve un resultado provisional:

[12]:

meta["spec"]["interim_results"] = {
    "$schema": "https://json-schema.org/draft/2019-09/schema",
    "description": "Parameter vector at current optimization step. This is a numpy array.",
    "type": "array",
}

Cargar el programa¶

Ahora tenemos todos los ingredientes necesarios para cargar nuestro programa. Para hacerlo, necesitamos recopilar todo nuestro código en un archivo, aquí llamado sample_vqe.py para ser cargado. Esta limitación se eliminará en versiones posteriores de Qiskit Runtime. Alternativamente, si el código completo está contenido dentro de una sola celda del cuaderno de Jupyter, esto se puede hacer usando la función mágica

%%writefile my_program.py

Para cargar realmente el programa, necesitamos inicializar un servicio Qiskit Runtime:

[13]:

from qiskit_ibm_runtime import QiskitRuntimeService

service = QiskitRuntimeService()

Carga del programa¶

La llamada a program_upload toma el archivo de Python objetivo como data y los metadatos como entradas.

[14]:

program_id = service.upload_program(data="sample_vqe.py", metadata=meta)
program_id

[14]:

'sample-vqe-E3DO04qqoQ'

Aquí el método upload_program() devuelve un program_id, que es como debes hacer referencia a tu programa.

Ejecutar el programa¶

Especificar parámetros¶

Para ejecutar el programa necesitamos especificar las options que se utilizan en el entorno de runtime (no las variables del programa). El backend de destino se define a través del campo backend_name.

Formatear los resultados devueltos¶

Para formatear los resultados devueltos en el formato deseado, necesitamos especificar un decodificador. Este decodificador debe tener un método decode que sea llamado para realizar la conversión real. En nuestro caso, OptimizeResult es una subclase simple de dict, por lo que el formato es sencillo.

[26]:

from qiskit_ibm_runtime.program import ResultDecoder
from scipy.optimize import OptimizeResult


class VQEResultDecoder(ResultDecoder):
    @classmethod
    def decode(cls, data):
        data = super().decode(data)  # This is required to preformat the data returned.
        return OptimizeResult(data)

Simplificación de la ejecución del programa con funciones contenedoras¶

Mientras que los programas runtime son poderosos y flexibles, no son las cosas más amigables para interactuar. Por lo tanto, si tu programa está destinado a ser utilizado por otros, es mejor crear funciones y/o clases contenedoras que simplifiquen la experiencia del usuario. Además, estos contenedores permiten la validación de las entradas del usuario en el lado del cliente, lo que puede encontrar rápidamente errores que, de otro modo, se generarían más tarde durante el proceso de ejecución, algo que podría haber llevado horas esperando en la cola para llegar.

Aquí haremos dos rutinas auxiliares. En primer lugar, un contenedor de trabajo que nos permite adjuntar y recuperar los resultados provisionales directamente desde el objeto de trabajo en sí, así como decodificar para nosotros, de modo que el usuario final no tenga que preocuparse por formatear los resultados.

[28]:

class RuntimeJobWrapper:
    """A simple Job wrapper that attaches interim results directly to the job object itself
    in the `interim_results attribute` via the `_callback` function.
    """

    def __init__(self):
        self._job = None
        self._decoder = VQEResultDecoder
        self.interim_results = []

    def _callback(self, job_id, xk):
        """The callback function that attaches interim results:

        Parameters:
            job_id (str): The job ID.
            xk (array_like): A list or NumPy array to attach.
        """
        self.interim_results.append(xk)

    def __getattr__(self, attr):
        if attr == "result":
            return self.result
        else:
            if attr in dir(self._job):
                return getattr(self._job, attr)
            raise AttributeError("Class does not have {}.".format(attr))

    def result(self):
        """Get the result of the job as a SciPy OptimizerResult object.

        This blocks until job is done, cancelled, or errors.

        Returns:
            OptimizerResult: A SciPy optimizer result object.
        """
        return self._job.result(decoder=self._decoder)

A continuación, creamos la función en sí que queremos que los usuarios llamen para ejecutar nuestro programa. A esta función, le agregaremos una serie de comprobaciones de validación sencillas (no todas las comprobaciones se realizarán por simplicidad), así como también usaremos el contenedor de trabajo definido anteriormente para simplificar la salida.

[29]:

import qiskit.circuit.library.n_local as lib_local


def vqe_runner(
    backend,
    hamiltonian,
    ansatz="EfficientSU2",
    ansatz_config={},
    x0=None,
    optimizer="SPSA",
    optimizer_config={"maxiter": 100},
    shots=8192,
    use_measurement_mitigation=False,
):

    """Routine that executes a given VQE problem via the sample-vqe program on the target backend.

    Parameters:
        backend (ProgramBackend): Qiskit backend instance.
        hamiltonian (list): Hamiltonian whose ground state we want to find.
        ansatz (str): Optional, name of ansatz quantum circuit to use, default='EfficientSU2'
        ansatz_config (dict): Optional, configuration parameters for the ansatz circuit.
        x0 (array_like): Optional, initial vector of parameters.
        optimizer (str): Optional, string specifying classical optimizer, default='SPSA'.
        optimizer_config (dict): Optional, configuration parameters for the optimizer.
        shots (int): Optional, number of shots to take per circuit.
        use_measurement_mitigation (bool): Optional, use measurement mitigation, default=False.

    Returns:
        OptimizeResult: The result in SciPy optimization format.
    """
    options = {"backend_name": backend.name}

    inputs = {}

    # Validate Hamiltonian is correct
    num_qubits = len(H[0][1])
    for idx, ham in enumerate(hamiltonian):
        if len(ham[1]) != num_qubits:
            raise ValueError(
                "Number of qubits in Hamiltonian term {} does not match {}".format(
                    idx, num_qubits
                )
            )
    inputs["hamiltonian"] = hamiltonian

    # Validate ansatz is in the module
    ansatz_circ = getattr(lib_local, ansatz, None)
    if not ansatz_circ:
        raise ValueError("Ansatz {} not in n_local circuit library.".format(ansatz))

    inputs["ansatz"] = ansatz
    inputs["ansatz_config"] = ansatz_config

    # If given x0, validate its length against num_params in ansatz:
    if x0:
        x0 = np.asarray(x0)
        ansatz_circ = ansatz_circ(num_qubits, **ansatz_config)
        num_params = ansatz_circ.num_parameters
        if x0.shape[0] != num_params:
            raise ValueError(
                "Length of x0 {} does not match number of params in ansatz {}".format(
                    x0.shape[0], num_params
                )
            )
    inputs["x0"] = x0

    # Set the rest of the inputs
    inputs["optimizer"] = optimizer
    inputs["optimizer_config"] = optimizer_config
    inputs["shots"] = shots
    inputs["use_measurement_mitigation"] = use_measurement_mitigation

    rt_job = RuntimeJobWrapper()
    job = service.run(
        program_id, options=options, inputs=inputs, callback=rt_job._callback
    )
    rt_job._job = job

    return rt_job

Conclusión¶

Hemos demostrado cómo crear, cargar y usar un Qiskit Runtime personalizado al crear nuestro propio solucionador VQE desde cero. Este tutorial estaba destinado a tocar todos los aspectos del proceso para un ejemplo del mundo real. Dentro de las limitaciones actuales del entorno de runtime, este ejemplo debería permitir a los lectores desarrollar su propio programa runtime de un solo archivo. Este programa también es un buen punto de partida para explorar versiones adicionales del runtime de VQE. Por ejemplo, es sencillo variar el número de iteraciones por repetición, aumentando las iteraciones a medida que aumenta el número de repeticiones. Aquellos que buscan profundizar pueden considerar implementar un VQE adaptativo, donde el ansatz no se fija en la inicialización.

Apéndice A¶

Aquí codificamos un optimizador simple de aproximación estocástica de perturbaciones simultáneas (simultaneous perturbation stochastic approximation, SPSA) para su uso en sistemas cuánticos ruidosos. La mayoría de los optimizadores no manejan bien las funciones de costos fluctuantes, por lo que esta es una adición necesaria para ejecutar en hardware cuántico real.

[4]:

import numpy as np
from scipy.optimize import OptimizeResult


def fmin_spsa(
    func,
    x0,
    args=(),
    maxiter=100,
    a=1.0,
    alpha=0.602,
    c=1.0,
    gamma=0.101,
    callback=None,
):
    """
    Minimization of scalar function of one or more variables using simultaneous
    perturbation stochastic approximation (SPSA).

    Parameters:
        func (callable): The objective function to be minimized.

                          ``fun(x, *args) -> float``

                          where x is an 1-D array with shape (n,) and args is a
                          tuple of the fixed parameters needed to completely
                          specify the function.

        x0 (ndarray): Initial guess. Array of real elements of size (n,),
                      where ‘n’ is the number of independent variables.

        maxiter (int): Maximum number of iterations.  The number of function
                       evaluations is twice as many. Optional.

        a (float): SPSA gradient scaling parameter. Optional.

        alpha (float): SPSA gradient scaling exponent. Optional.

        c (float):  SPSA step size scaling parameter. Optional.

        gamma (float): SPSA step size scaling exponent. Optional.

        callback (callable): Function that accepts the current parameter vector
                             as input.

    Returns:
        OptimizeResult: Solution in SciPy Optimization format.

    Notes:
        See the `SPSA homepage <https://www.jhuapl.edu/SPSA/>`_ for usage and
        additional extentions to the basic version implimented here.
    """
    A = 0.01 * maxiter
    x0 = np.asarray(x0)
    x = x0

    for kk in range(maxiter):
        ak = a * (kk + 1.0 + A) ** -alpha
        ck = c * (kk + 1.0) ** -gamma
        # Bernoulli distribution for randoms
        deltak = 2 * np.random.randint(2, size=x.shape[0]) - 1
        grad = (func(x + ck * deltak, *args) - func(x - ck * deltak, *args)) / (
            2 * ck * deltak
        )
        x -= ak * grad

        if callback is not None:
            callback(x)

    return OptimizeResult(
        fun=func(x, *args),
        x=x,
        nit=maxiter,
        nfev=2 * maxiter,
        message="Optimization terminated successfully.",
        success=True,
    )

Apéndice B¶

Esta es una función auxiliar que convierte los operadores de Pauli en las cadenas que definen los operadores Hamiltonianos en las mediciones apropiadas al final de los circuitos. Para los operadores \(X\), esto implica agregar una compuerta \(H\) a los qubits que se van a medir, mientras que un operador \(Y\) necesita un \(S^{+}\) seguido de una \(H\). Otras opciones de operadores de Pauli no requieren compuertas adicionales antes de la medición.

[5]:

def opstr_to_meas_circ(op_str):
    """Takes a list of operator strings and makes circuit with the correct post-rotations for measurements.

    Parameters:
        op_str (list): List of strings representing the operators needed for measurements.

    Returns:
        list: List of circuits for measurement post-rotations
    """
    num_qubits = len(op_str[0])
    circs = []
    for op in op_str:
        qc = QuantumCircuit(num_qubits)
        for idx, item in enumerate(op):
            if item == "X":
                qc.h(num_qubits - idx - 1)
            elif item == "Y":
                qc.sdg(num_qubits - idx - 1)
                qc.h(num_qubits - idx - 1)
        circs.append(qc)
    return circs
