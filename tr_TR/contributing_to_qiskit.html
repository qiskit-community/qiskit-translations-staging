Koda Katkı Sağlama¶

Stil Rehberi¶

Projede tutarlı bir kod stili uygulamak için kod katkılarının projenin stil rehberine uyduğunu ve saygı duyduğunu onaylamak için Pylint ve pycodesytle kullanırız. Değişikliklerinizin stil rehberine uyduğunu doğrulamak için tox -elint komutunu çalıştırın

Katılımcı Lisans Sözleşmesi¶

Herhangi bir kod göndermeden önce, bütün katılımcılar Katılımcı Lisans Sözleşmesi(CLA)’ni imzalamak zorunda. Bir CLA imzalayarak, katkının yazarının siz olduğunuzu ve Apache-2.0 lisansının koşullarına göre özgür bir şekilde katkı sağladığınızı tasdik edersiniz.

Yeni bir pull request ile Qiskit projesine katkı sağladığınızda, bir bot CLA’yı imzalayıp imzalamadığınızı değerlendirecek. Eğer gerekirse, bot anlaşmayı kabul etmek için bir link de dahil olmak üzere pull request hakkında yorum yapacak. Bireysel CLA belgesi PDF olarak incelenmeye uygundur.

Not

Eğer katkınız işinizin parçası ya da işvereninizin mülkiyeti ise, o zaman büyük ihtimalle Kurumsal CLA de imzalamanız ve onu <qiskit@us.ibm.com> üzerinden bize email ile göndermeniz gerekecektir.

Pull Requestler¶

We use GitHub pull requests to accept contributions.

Pull request’i açmadan önce üzerinde çalıştığınız özellik veya çözdüğünüz hata hakkında gerekli olmasa da yeni bir sorun açmak, çalışmanız hakkında topluluk ile bir tartışma başlatmada önemli bir adımdır. Sorun fikir ve onu koda uygulamak için birlikte nasıl çalışacağımız hakkında konuşmamız için bize bir yer verir. Ayrıca topluluğun sizin ne üzerine çalıştığınızı bilmesine izin verir ve eğer yardıma ihtiyacınız olursa diğer topluluk ve takım üyeleri ile tartışırken sorunu kaynak gösterebilirsiniz.

Eğer biraz kod yazdıysanız fakat bitirmede yardıma ihtiyacınız varsa, bitirmeden önce başlangıç feedback i almak veya kodu paylaşmak ve uygulamayı tamamlamadan tartışmak istiyorsanız, Devam Eden Çalışma (Work in Progress) pull request’i açabilirsiniz. Pull request’i oluştururken başlığı başına [WIP] (Work In Progress için) etiketi ile ekleyin. Bu görüntüleyenlere(reviewers), PR’teki kodun son halinde olmadığını ve değişeceğini gösterecektir. Bu ayrıca tamamlanana dek öneriyi birleştirmeyeceğimiz anlamına gelir. Kod birleştirmek için tamamen görüntülenmeye hazır olduğunda siz ya da bir görüntüleyen(reviewer) [WIP] etiketini kaldırabilir.

Commit Mesajları¶

Bir değişikliği açıklayan öneri mesajının içeriği, değişikliğin kendisi kadar önemlidir. Öneri mesajı sadece kodu gözden geçirmeye değil ayrıca git logdaki değişiklik geçmişine de yarar sağlar. Detaylı bir öneri mesajı kodunuzun gözden geçirilmesini daha kolay hale getirecek ve gelecekte birisi koda baktığında değişim ortamına yarar sağlayacaktır. Bir öneri mesajı yazarken şu önemli detayları aklınızda tutun:

Görüntüleyenin asıl sorunun ne olduğunu bildiğini varsaymayın.

Bir sorunu okurken aşağı yukarı bir dizi yorumdan sonra problemin temelindeki sebebin ne olduğu genellikle net olur. Commit mesajları, özgün sorunun ne olduğu konusunda net bir ifadeye sahip olmalıdır. Hata sorunun nasıl tanımlandığının sadece ilginç bir tarihsel arka planıdır. Hata etiketini okumaya ihtiyaç olmadan commit mesajlarından doğruluk için önerilen yamaya göz atmak mümkün olmalıdır.

Kodun aşikar/kendini belgeleyen olduğunu düşünmeyin.

Bir kişi için apaçık olan, başka bir kişi için açık olmayabilir. En bariz yazım hataları veya sadece boşluklarla ilgili işlemler dışındaki tüm değişiklikler için her zaman orijinal sorunun ne olduğunu ve nasıl düzeltildiğini belgeleyin.

Neden bir değişiklik yapıldığını ifade et.

Yaygın bir hata, neden geliştiricinin o şekilde yapmayı seçtiğini tanımlamadan sadece kodun nasıl yazıldığını belgelemektir. Kesinlikle, özellikle büyük değişiklikler için bütün kod yapısını tanımlamalısınız fakat daha da önemlisi, değişikliklerin arkasındaki maksadı/niyeti tanımladığınıza emin olmalısınız.

Gelişmiş kod yapısının ima edilip edilmediğini anlamak için commit mesajını okuyun.

Genellikle büyük bir commit mesajını tanımlarken commitin iki ya da daha fazla parçaya ayrılması aşikar olur. Değişikliğin özel pull requestlerin içine bölünmesi için geri dönmekten ve değişikliği rebase (bir referanstan diğerine dayalı olarak verileri değiştirmek) etmekten korkmayın.

Gözden geçirip geçirmemeye karar vermek için yeterli bilgi sağlayın.

GitHub yeni pull request teklifleri için e-mail uyarıları gönderdiğinde, genellikle sadece commit mesajını ve değişen dosyaların listesini içeren küçük bir bilgilendirme vardır. Yüksek hacimli yamalar yüzünden, potansiyel reviewerların görüntülemeleri gereken yamayı bulmaları için bir commit mesajı yeterli bilgiyi içermelidir.

İlk commit satırı en önemlisidir.

Git commitlerinde, commit mesajının ilk satırının özel değeri vardır. Önerilen pull request etiketi, e-mail bildirimi konu satırı, git açıklayıcı mesajları, gitk görüntüleyici ek açıklamaları, merge commit mesajları ve alanın gözde olduğu birçok yer olarak kullanılır. Değişim kendisini özetleyebildiği kadar kodun hangi kısmının etkilendiğinin detaylandırılmasına da önem vermelidir.

Ek olarak, eğer PR changelogun içerisine dahil edilerek etiketlendiyse commit mesajının ilk satırı, üretilen changelog içerisinde bir giriş haline gelir. Açık, kısa ve öz özet satırları yazmanız son derece önemlidir.

Geçerli kodun her sınırlandırmasını tanımlayın.

Eğer değiştirilen kod gelişimler ya da bilinen herhangi bir sınırlandırma için hala gelecek faaliyet alanlarına sahipse, bunları commit mesajında belirtin. Bu reviewerın daha kapsamlı bir resim belirlemesini sağlar ve uzun zamanlı isteklerle kısa zamanlı istekleri kıyaslandığında hangi takasların yapılacağını gösterir.

Sorunlara referanslar ekleyin.

Eğer commit onarımları bir sorun ile ilişkiliyse, bunu commit mesajında ek olarak açıkladığınızdan emin oldun. Bu sintaksı kullanın:

Fixes #1234

eğer sorunu çözerse (PR birleştirildiğinde GitHub sorunu kapatacak).

Takip edilecek temel kural:

Commit mesajı doğruluğa uygun yamayı tamamen anlamak ve gözden geçirmek için gerekli tüm bilgiyi içermelidir. Daha azı iyi değildir.

Kodunuzu Belgelemek¶

Eğer bir ögeye değişiklik yapmak istiyorsanız uyumlu repoda docs/apidocs altındaki dokümantasyonun parçalarını ve ilişkilendirilmiş *docstrings*i güncellediğinizden emin olun. Yerel olarak dokümantasyona özel öge oluşturmak, yerel olarak dokümantasyon inşa edip derlemek ve çıktıyı docs/_build/html a kaydetmek için tox -edocs u çalıştır. Ek olarak, Docs Cl işi azure pipelineları üzerinde bunu çalıştıracak ve yerel olarak yükleyip görüntüleyebileceğiniz çıktının zip dosyasını barındıracaktır.

Eğer Qiskit/qiskit repo’de tutulan combined documentation (birleştirilmiş dokümantasyon) ile bir sorununuz varsa, belge hataları görüyorsanız, belgelenmesi gereken yeni bir özellik varsa ya da bu materyalin var olan dosyalara eklenmesi gerektiğini düşünüyorsanız yeni bir dokümantasyon sorunu açabilirsiniz <https://github.com/Qiskit/qiskit/issues/new/choose>`__.

İlk Yararlı Katkılar¶

Qiskit’e katkı sağlamak istiyorsanız fakat nereden başlayacağınıza emin değilseniz projeyle ilgili konulardaki good first issue (ilk yararlı konu) etiketi projeye yeni katılan kişiler için uygun ögeleri vurgular. Bunlar katkıda bulunanlar tarafından incelenmiş ve etiketlenmiş yeni katkı sağlayanın üzerine çalışabilir olması gereken bütün konulardır. Diğer kelimelerde, soruna uygun bir düzeltme geliştirmek için Qiskit ile samimi aşinalık gerekli değildir.

Kullanım Dışı Bırakma Bildirimi¶

Qiskit kullanıcıları güvendikleri bir özellik ya da API’ın yarın yazılım tarafından desteklenip desteklenmeyeceğini bilmelidir. Hangi koşullar altında projenin bir özellik ya da API’ı kaldırabileceğini (ya da geriye dönük uyumsuz bir şekilde değiştirebileceğini) bilmek kullanıcı için önemlidir. Aşağıdaki bildiri beklentileri idare etmek için API’ın ve özelliğin kullanım dışı bırakılmasının/kaldırılmasının Qiskit tarafından nasıl yürütüldüğüdür:

1. Features, APIs, or configuration options are marked deprecated in the code. Appropriate DeprecationWarning class warnings will be sent to the user. The deprecated code will be frozen and receive only minimal maintenance (just so that it continues to work as-is).

2. A migration path will be documented for current users of the feature. This will be outlined in the both the release notes adding the deprecation, and the release notes removing the feature at the completion of the deprecation cycle. If feasible, the warning message will also include the migration path. A migration path might be “stop using that feature”, but in such cases it is necessary to first judge how widely used and/or important the feature is to users, in order to determine a reasonable obsolescence date.

3. An obsolescence date for the feature will be set. The feature must remain intact and working (although with the proper warning being emitted) in all releases pushed until after that obsolescence date. At the very minimum, the feature (or API, or configuration option) should be marked as deprecated (and continue to be supported) for at least three months of linear time from the release date of the first release to include the deprecation warning. For example, if a feature were deprecated in the 0.9.0 release of Terra, which was released on August 22, 2019, then that feature should still appear in all releases until at least November 22, 2019. Since releases do not occur at fixed time intervals, a deprecation warning may only occur in one release prior to removal.

Bu gecikmenin en az olduğunu not edin. Önemli özellikler için, kullanım dışı özelliğin en az bu gecikme zamanının iki katı için görünmesi tavsiye edilir. Ayrıca kararlı dallanma politikası gereğince, kullanımdan kaldırma sadece daha küçük sürümlerde olabilir; backporting (bir yazılımda daha yeni versiyonda geliştirilmiş olan bir özelliğin eski versiyonlara taşınması) için uygun değillerdir.

Kullanımdan Kaldırma Uyarıları¶

Kullanımdan kaldırma uyarısı vermenin uygun yolu, Python standart kütüphanesinde warnings modülünden `` warn`` işlevini kullanmaktır. Uyarı kategorisi sınıfı bir DeprecationWarning olmalıdır. Örnek olarak şu verilebilir:

import warnings

def foo(input):
    warnings.warn('The qiskit.foo() function is deprecated as of 0.9.0, and '
                  'will be removed no earlier than 3 months after that '
                  'release date. You should use the qiskit.bar() function '
                  'instead.', DeprecationWarning, stacklevel=2)

Burada dikkat edilmesi gereken şey, warn() çağrısında stack_level anahtar kelime argümanıdır. Bu argüman, uyarıyı başlatan satır olarak çağrı yığınında hangi seviyenin kullanılacağını belirlemek için kullanılır. Genellikle, stack_level, 2 olarak ayarlanmalıdır, çünkü bu, uyarının oluşturulduğu içeriği çağıran satırı gösterecektir. Yukarıdaki örnekte, bu foo() ‘nun çağırıcısı olacaktır. Bunu ayarlamadıysanız, uyarı foo() işlevindeki satırdan kaynaklandığını gösterir; bu, kullanımdan kaldırılmış bir çağrının kaynağını belirlemeye çalışan kullanıcılar için yararlı değildir. Bununla birlikte, bu değer, çağrı yığınına ve warn() öğesinin nereden çağrıldığına bağlı olarak ayarlanabilir. Örneğin, uyarı her zaman yalnızca bir çağıranı olan bir özel metod ile verilirse, stack_level = 3 uygun olabilir.

Kararlı Dallanma Politikası¶

Kararlı dallanmada yüksek etkili hatalara ve yayınlandığından beri temel üzerinde düzeltilen güvenlik sorunlarına uygun güvenli bir onarımlar kaynağı olması istenir. Bir kararlı dal PR’ini görüntülediğimizde, kararlı şubelerin kullanıcılarına yarar sağlayacak değerle birlikte her verilen yamanın riskini dengelemeliyiz. Sadece sınırlı bir değişiklikler sınıfının kararlı dallanmaya alınması uygundur. Büyük bir sorun için büyük riskli bir yama, belli belirsiz bir hata için ufak bir onarımın uygulanması mantıklı olabilir. Bir değişikliğe karar verirken bir dizi etken dikkate alınmalıdır:

  • Gerileme riski: en zayıf değişiklikler bile bir şeyleri biraz bozma riski taşır ve biz kararlı dalda gerilemeden gerçekten kaçınmak isteriz.

  • Kullanıcı görünürlük yararı: onardığımız şeyler aslında kullanıcıların farkına varabildiği şeyler mi, öyle olsa bile ne kadar önemli?

  • Onarım ne kadar kendi kendine yeterli: eğer belirli bir sorunu onarıyor fakat bütün kodu yeniden etkiliyorsa, daha az riskli bir onarımın nasıl olacağı üzerinde düşünmeye muhtemelen değer.

  • Düzeltme zaten ana kodlarda olsun ya da olmasın: değişiklik basitçe ana kodlar üzerinde bir anlam ifade etmediği sürece, bir değişiklik ana birimle zaten birleştirilmiş olan bir değişikliğin önceki versiyonuna taşınmalıdır.

Backport(Eski versiyona güncel özellik taşıma) prosedürü:¶

Bir yamayı master’dan kararlı eski sürüme eklerken, master’daki değişikliğe dair bir referans tutmak istiyoruz. Kararlı PR (Pull Request) için dallanma oluşturduğunuzda, şunu kullanın:

$ git cherry-pick -x $master_commit_id

Ancak, bu sadece master’ın kendi kendine yeten küçük yamaları için çalışır. Daha büyük değişikliğin bir alt kümesini (örneğin ezilmiş bir PR’dan) master’dan aktarmanız gerekiyorsa, bunu elle (manuel olarak) yapınız. Bu durumlarda şunu ekleyin:

Backported from: #master pr number

böylece, sıkı bir cımbızlama mantıklı olmasa bile, değişiklik alt kümesinin kaynağını takip edebiliriz.

Önerdiğiniz yama temiz bir şekilde cımbızlanmazsa, çatışmaları kendiniz çözerek ve ortaya çıkan yamayı önererek yardımcı olabilirsiniz. Kararlı yamanın incelenmesine yardımcı olmak için lütfen Çakışma satırlarını taahhüt mesajında tutun.

Backport etiketleri¶

kararlı backport potansiyeli ile etiketlenmiş hatalar veya PR’lar, kararlı sürüm için de geçerli olan hatalardır ve bir master’a sabitlendikten sonra backporting için uygun olabilirler. Backport önerildikten sonra etiket kaldırılmalıdır.

Hedef dalın kararlı olarak ayarlanmasının bir hata olmadığını gösteren bir işaret olarak kararlı dalda PR başlığına [Stable] ekleyin. Ayrıca, taşıdığınız masterdaki PR numarasını referans alın.

Belgelemeye katkı sağlamak¶

Qiskit dokümantasyonu, ağırlıklı olarak style of API documentation (API dokümantasyon stili)’ndeki Qiskit kod yorumlarından çekilen docs as code (kod olarak dokümanlar) felsefesi ile şekillendirilmektedir.

Dokümantasyon, Sphinx kullanılarak Qiskit/qiskit/docs ana dalından (master branch) oluşturulmuştur. API Reference altındaki dokümantasyonun büyük bir kısmı Kodlar Nerede içerisinde listelenen repolardaki kod yorumlarından çekilmiştir.

Belgeleme yapısı¶

Qiskit’te belgenin yapılandırılma şekli, asıl belgenin mümkün olduğunca belge dizgisine aktarılmasıdır. Bu, belgelerin çoğu değiştirilen kodun yanında bulunduğu için geliştirme sürecinde ekleme ve düzeltmeler yapılmasını kolaylaştırır. Terra’nın normal belge yapısında üç seviye vardır:

“docs/apidocs” içindeki “.rst” dosyaları

Bu belgeler, Sphinx’e işlenen belgelere hangi modüllerin dahil edileceğini söylemek için kullanılmaktadır. Bu iki bilgi parçası içerir: dokümantasyon içerisinde dahili linkler için kullanılabilecek modüle bir (dahili referans) internal reference ya da (çapraz referans)` cross reference <https://www.sphinx-doc.org/en/latest/usage/restructuredtext/roles.html#ref-role>`__ ve modül docstringlerini, belirtilen import pathinden ayırmak için kullanılan bir (otomodül direktifi) automodule directive. Örneğin; dagcircuit.rst dosyası şunları içerir:

.. _qiskit-dagcircuit:


.. automodule:: qiskit.dagcircuit
   :no-members:
   :no-inherited-members:
   :no-special-members:

Bunun dışındaki tek .rst dosyası, içindekiler tablosunu içeren qiskit.rst dosyasıdır. Eğer yeni bir modülün dokümantasyonu için yeni bir .rst dosyası ekliyorsanız onu bu klasördeki toctree toctree ‘ye eklediğinizden emin olun.

Modül düzeyinde docstring

Bu docstring rst dosyasındaki automodule direktifinde belirtilen modüle uygun modül düzeyindedir. Eğer modül dizin/namespace olarak belirtilmişse o dizin için docstring __init__.py içerisinde belirtilmelidir. Bu modül düzeyindeki docstring belgenen modülün hakkında daha fazla detay içermektedir. Bu docstring için normal yapı, o modülün içerdiği genel (public) API’ın bütün sınıflarını ve fonksiyonlarını çerçevelemektir. Bu, genel olarak eğer modül bu qiskit.execute olayındaki gibi basitse (otomatik özet direktifi) autosummary directive ya da (otomatik belge direktifleri) autodoc directives kullanarak doğrudan yapılır. Otomatik özet direktifi her biri için elle (manuel olarak) autodoc direktifi çağırmaya gerek kalmadan doğrudan farklı Python birimlerinin (sınıflar, fonksiyonlar, vb.) listesini autodoc için kullanılmaktadır. Modül düzeyinde docstring, modülün hangi fonksiyonelliği sağladığı konusunda üst düzey bir bakış sağladığı yerdir. Bu normalde genel (public) API’nın farklı bileşenlerini birden fazla alt bölüme gruplayarak yapılır.

Örneğin, önceki dagcircuit modülü örneğinde olduğu gibi qiskit/dagcircuit/__init__.py için modül docstringinin içeriği aşağıdaki gibi olmalıdır:

"""
=======================================
DAG Circuits (:mod:`qiskit.dagcircuit`)
=======================================
.. currentmodule:: qiskit.dagcircuit
DAG Circuits
============
.. autosummary::
   :toctree: ../stubs/
   DAGCircuit
   DAGNode
Exceptions
==========
.. autosummary::
   :toctree: ../stubs/
   DAGCircuitError
"""

Not

Bu sadece bir örnektir ve dagcicuit modülü için asıl modül docstringi bundan farklı olabilir.

Elementler için asıl docstring, modül docstringi içerisinde listelenir

Gerektiğinde örnekler kullanarak ortaya çıkarılan bütün genel arayüzleri (public interfaces) tamamen belgelemek için çabalamalısınız. Doctringler için Google Python Style Docstrings kullanılmaktadır. Bu, napoleon sphinx extension kullanılarak ayrılır. Napoleon dokümantasyonu napoleon documentation “Docstringler nasıl biçimlendirilmelidir?” in güzel bir örneğini içermektedir.

Not

Mantıklı olması için bir docstringde herhangi bir Sphinx direktifi ya da rst biçimlendirmesi kullanabilirsiniz. Örneğin, kullanılan yaygın bir uzantı, Jupyter’de bir kod blogu çalıştırıp hem kodu hem de çıktıyı ekrana yazdırmak için kullanılan jupyter-execute direktifidir. Bu özellikle görselleştirme için kullanışlıdır.

Dokümantasyon Entegrasyonu¶

Qiskit’in dökümantasyon web sayfası (https://qiskit.org/documentation/) adresinde olan dokümantasyon Qiskit Projesi’nin tamamını kapsar; Terra sadece bunun bir bileşenidir. Bunun gibi, barındırılan sürüm için dokümantasyon yapıları Qiskit meta-paket deposu/reposu (https://github.com/Qiskit/qiskit) tarafından yapılır. Taahhütler bu repo ile birleştirildiğinde Sphinx yapılarının çıktısı qiskit.org web sayfasına yüklenir. Bu Sphinx yapıları oradaki meta-paket tarafından yüklenen Qiskit elementlerinin sürümünden dokümantasyona çekilmesi için konfigüre edilir. Örneğin, meta-paket versiyonu şu anda 0.13.0, o zaman bu Terra’nın 0.10.0 sürümünden belgeye kopyalanacaktır. Meta-paketlerin gereksinimleri temizlendiğinde, o zaman dokümantasyonu yeni sürümünden çekmeye başlayacak. Bu doğru olmayan API dokümantasyonunun düzeltmelerinin yeni bir sürüme dahil edilmesi gerekeceği anlamına gelir. Dokümantasyon düzeltmeleri, kararlı dallanma başına kararlı yama sürümü politikası için geçerli backportlar (yeni özellikleri eski versiyona taşıma) şeklindedir (aşağıdaki bölüme bakınız.).

İnşa sürecisi boyunca, her elementin içeriği docs/apidocs/ tekrarlı bir şekilde diğer tüm elementlerle birlikte meta-paket reposundaki doc/apidocs/ un paylaşılan bir kopyasına kopyalanır. Bu bir sürümdeki her elementte docs/apidocs un kökünde bulunanların https://qiskit.org/documentation/apidoc/ kökünde sonlanacağı anlamına gelir.

Belgelerin Çevrilmesi¶

Qiskit dokümantasyonu, organizasyonların çeviri projeleri koordine etmesini ve materyalleri çevirmek için topluluklarla iş birliği yapmasını sağlayan bir yazılım ve web yerelleştirme platformu olan Crowdin kulllanılarak çevrilir (yerelleştirilir). Crowdin tercümanlar topluluğu tarafından oluşturulan bir cümle çevirisi işini diğer benzer cümleleri çevirmek için otomatik olarak tekrar kullanarak tercümanlarımızın etkisini güçlendirmek için olanak sağlamaktadır. Crowdin ayrıca çevirilerde dosyalar arasında cümleleri hareket ettirmek gibi orijinal materyalde yapılan birçok türde değişikliğe dayanıklı hale getirir.

Qiskit yerelleştirme istekleri Qiskit Translations repository/deposuna işlenir. Qiskit yerelleştirmesine katkı sağlamak için, lütfen bu adımları takip edin:

  1. İsminizi ya da (ID) LOCALIZATION_CONTRIBUTORS dosyasına ekleyin.

  2. Değişikliğinizi birleştirmek için bir pull request (PR) oluşturun. Bir pull request açmak için şablonu takip ettiğinizden emin olun.

    Not

    • Her katkıda bulunan kişi kendi PR’ini oluşturmak ve CLA’yı imzalamak zorundadır.

    • Lütfen PR özetinde katkı sağlamak istediğiniz dili belirtin.

    • Eğer bir dil isteği için açık bir sorununuz varsa, PR’e sorun bağlantısını ekleyin.

  3. Qiskit Katkı Sağlayan Lisans Sözleşmesi’ni (CLA) imzalamanız istenecektir; lütfen bunu yapın.

  4. Herhangi bir yeni dilin eklenmesinde, yerelleştirme projesinin yöneticilerinden resmi destek alabilmesi için bir dile en az katkıda bulunan üç kişi gereklidir.

  5. Katkıda bulunan kişilerin grubu arasında, yerelleştirme projesinin yöneticileriyle irtibatta olacak kişi olarak hizmet edecek bir tercüman liderinin tanımlanması gerekir. Liderin email yolu ile iletişime geçmesi gereklidir: Yuri Kobayashi (yurik@jp.ibm.com).

    Not

Eğer bir tercüman lideri olmak istiyorsanız ya da yeni bir çeviri ekibine katılmaya istekli olacaksınız, Qiskit ekibi ile görüşme başlatmak ve çeviri ekibi üyelerini güçlendirmek için bir GitHub issue açabilirsiniz.

  7. Join(Katıl) butonuna tıklayın ve neden Crowdin projesine katılmak istediğinizin sorulduğu yerdeki diyalog kutusuna PR’inizin URL’sini yapıştırın.

Crowdin projesi yöneticileri sizin isteğinizi gözden geçirecek ve size verebildikleri kadar çabuk bir şekilde erişim verecekler.

Kaynaktan oluşturma¶

Qiskit/qiskit havuzunda yer alan, yerel kopyanızdan belgelerin yerel bir kopyasını aşağıdaki gibi oluşturabilirsiniz:

  1. Qiskit deposunu klonla.

    git clone https://github.com/Qiskit/qiskit.git

  2. Depoyu klonlamak, qiskit adında yerel bir dosya oluşturur.

    cd qiskit

  3. Qiskit/qiskit yerel kopyanıza giderek dokümantasyonu oluşturun ve terminal penceresinde aşağıdaki komutu çalıştırıp oluşturun.

    tox -edocs

    Eğer hali hazırda tox komutu yüklenmemişse, şununla kurulumu yapın:

    pip install tox

Eğer yerel RST dosyalarınızda değişiklik yaparsanız, /doc/ adresine giderek ve bir terminal penceresinde aşağıdakileri çalıştırarak HTML dosyalarınızı güncelleyebilirsiniz:

tox -edocs

Bu işlem, /docs/_build/html/ altdizininde yerel belge repository/deponuzun şekillendirilmiş bir HTML sürümünü oluşturacaktır.

Kaynaktan Terra Kurulumu¶

Bu kaynaktan kurulum yapılabilmesi için sisteminiz C++11’i destekleyen bir C++ derleyicisi olması gerekmektedir.

Çoğu Linux platformunda gerekli GCC derleyicisi zaten kuruludur.

MacOS kullanıyorsanız, XCode kurarak Clang derleyicisini kurabilirsiniz. Bir uçbirim penceresi açarak XCode ve Clang in kurulu olup olmadığını denetlemek için aşağıdakini giriniz.

clang --version

Takip eden komutu kullanarak XCode ve Clang’ı kurun.

xcode-select --install

Windows’ta, Visual C++ derleyicisini Visual Studio 2017 için Derleme Araçları. Bunun yerine C++ derleyicisini yükleme seçeneklerini belirlediğinizden emin olarak Visual Studio sürüm 2015 veya 2017’yi yükleyebilirsiniz.

Derleyicileri kurulduktan sonra, Qiskit Terra’yı kurmaya hazırsınız.

  1. Terra deposunu klonlayın.

    git clone https://github.com/Qiskit/qiskit-terra.git

  2. Depoyu klonlamak qiskit-terra adında yerel bir dosya oluşturur.

    cd qiskit-terra

  3. Python gereksinim kütüphanelerini qiskit-terra dizininizden kurun.

    pip install cython

  4. Test veya linting (tiftikleme) kontrolleri yapmak istiyorsanız, geliştirici gereksinimlerini yükleyin.

    pip install -r requirements-dev.txt

  5. qiskit-terra yükleyin.

    pip install .

Düzenlenebilir modda yüklemek istiyorsanız, yani projedeki kod değişikliklerinin yeniden yüklenmesi gerekmiyorsa, bunu şu şekilde yapabilirsiniz:

pip install -e .

Terra kurulumundan sonra kod örneklerini çalıştırabilirsiniz. Örneği aşağıdaki komutla çalıştırabilirsiniz.

python examples/python/using_qiskit_terra_level_0.py

Not

Başka bir bileşen kurmak istemiyorsanız, qiskit-terra, hem qiskit-aer hem de qiskit-ibmq sağlayıcı yüklü olmadığı konusunda bir RunTimeWarning (Çalışma Zamanı Uyarısı) uyarısı verecektir. Bu, kullanıcılar genellikle ek öğeleri kullanmayı düşündükleri, ancak yüklü olmadıklarının farkında olmadıkları veya Aer veya IBM Kuantum Sağlayıcısı’nın kurulumunun bir nedenden dolayı başarısız olduğu için yapılır. Bu uyarıları gizlemek istiyorsanız, kodunuza şunu ekleyin:

import warnings
warnings.filterwarnings('ignore', category=RuntimeWarning,
                        module='qiskit')

bunu kodunuza herhangi bir qiskit aktarmadan önce yapın. Bu, eksik qiskit-aer ve qiskit-ibmq sağlayıcısı hakkındaki uyarıyı bastırır, ancak qiskit veya diğer paketlerden gelen diğer uyarıları göstermeye devam eder.

Kaynaktan Aer Kurulumu¶

  1. Aer deposunu klonla.

    git clone https://github.com/Qiskit/qiskit-aer

  2. Oluşturma gereksinimlerini kur.

    pip install cmake scikit-build cython

Bundan sonra, Aer’i yükleme adımları kullandığınız işletim sistemine bağlıdır. Aer, Python arabirimine sahip derlenmiş bir C++ programı olduğundan, işletim sistemine bağlı olarak evrensel olarak yüklenemeyen Aer ikili dosyasını oluşturmak için Python olmayan bağımlılıklar vardır.

  1. Derleyici gereksinimlerini yükleyin.

    Aer’i kurmak bir C++ derleyicisi ve geliştirme başlıklarını gerektirir.

    Fedora veya ona eşdeğer bir Linux dağıtımı kullanıyorsanız aşağıdakileri özellikleri kullanarak yükleyin:

    dnf install @development-tools

    Ubuntu/Debian için şunu kullanarak kurun:

    apt-get install build-essential

  2. OpenBLAS geliştirme üstbilgilerini kurun.

    Fedora veya ona eşdeğer bir Linux dağıtımı kullanıyorsanız aşağıdakileri özellikleri kullanarak yükleyin:

    dnf install openblas-devel

    Ubuntu/Debian için şunu kullanarak kurun:

    apt-get install libopenblas-dev

  1. Bağımlılıkları kurun.

    MacOS’ta Clang derleyicisini kullanmak için OpenMP ‘i destekleyen fazladan bir kitaplık yüklemeniz gerekir. Bunu ve diğer bağımlılıkları yüklemek için brew kullanabilirsiniz.

    brew install libomp

  2. Sonra bir BLAS uygulaması yükleyin; OpenBLAS <https://www.openblas.net/> __ varsayılan seçimdir.

    brew install openblas

    Daha sonra, Xcode Command Line Tools programını yükleyin.

    xcode-select --install

Windows’ta tüm bağımlılıkları yüklemek için Anaconda3 veya Miniconda3 kullamanız gerekir.

  1. Derleyici gereksinimlerini yükleyin.

    conda install --update-deps vs2017_win-64 vs2017_win-32 msvc_runtime

  2. Kurulum ve derleme için gereklilikleri yükleyiniz.

    conda install --update-deps -c conda-forge -y openblas cmake

  1. Doğrudan qiskit-aer derle ve kur

    Sisteminizde pip < 19.0.0 kuruluysa ve ortamınız özel bir oluşturma gerektirmiyorsa, şu işlemi çalıştırın:

    cd qiskit-aer
pip install .

    Bu, ikili dosyaları oluşturur ve Aer ‘i kurar.

    Alternatif olarak, daha yeni bir pip’iniz varsa veya bazı özel gereksinimleriniz varsa, manuel olarak bir Python çarkı oluşturabilirsiniz.

    cd qiskit-aer
python ./setup.py bdist_wheel

    Çark oluşturma sırasında özel bir seçenek ayarlamanız gerekirse, buraya bakınız Çark oluşturma sırasında özel seçenekler.

    Python çarkını oluşturduktan sonra, Aer deposundaki dist/ dizininde saklanacaktır. Tam sürümü şuna bağlıdır,

    cd dist
pip install qiskit_aer-*.whl

    Çıktı çarkı dosyasının tam dosya adı, geliştirilmekte olan Aer’in geçerli sürümüne bağlıdır.

Çark oluşturma sırasında özel seçenekler¶

Aer derleme sistemi derlemeyi Python arabirimi ile derlerken çalıştırmak için scikit-build kullanır. CMake ‘i çağırma ve yerel sisteminiz için ikili dosyaları derlemede `kurulum araçları <https://setuptools.readthedocs.io/en/latest/>__ için bir arayüz görevi görür.

İkili dosyaları derlemenin karmaşıklığı nedeniyle, seçenekleri derleme sürecinin belirli bir bölümüne geçirmeniz gerekebilir. Değişkenleri geçirmenin yolu şudur:

python setup.py bdist_wheel [skbuild_opts] [-- [cmake_opts] [-- build_tool_opts]]

köşeli parantez [] içindeki öğeler isteğe bağlıdır ve skbuild_opts, cmake_opts, build_tool_opts, seçtiğiniz bayraklarla değiştirilir. *CMake * seçeneklerinin bir listesini burada bulabilirsiniz: https://cmake.org/cmake/help/v3.6/manual/cmake.1.html#options. Örneğin, şöyle bir şey çalıştırabilirsiniz:

python setup.py bdist_wheel -- -- -j8

Bu, -j8 bayrağını alttaki oluşturma sistemine geçirir (bu durumda Automake), 8 işlem kullanarak paralel olarak oluşturmak istediğinizi söyler.

Örneğin, linux üzerindeki bu bayraklar için yaygın bir kullanım durumu, kullanılacak C++ derleyicisinin belirli bir sürümünü belirtmektir (normalde varsayılan çok eski ise):

python setup.py bdist_wheel -- -DCMAKE_CXX_COMPILER=g++-7

bu da CMake’e Aer derlenirken varsayılan g++ yerine g++-7 komutunu kullanmasını söyleyecektir.

Bunun için bir diğer yaygın kullanım durumu da, ortamınıza bağlı olarak platform adınızı belirtmeniz ve statik bağlantıyı kapatmanız gerekebileceğidir.

python setup.py bdist_wheel --plat-name macosx-10.9-x86_64 \
-- -DSTATIC_LINKING=False -- -j8

Burada --plat-name, paket meta verilerinde kullanılacak platform adını belirtmek için, kurulum araçlarına bir bayraktır, -DSTATIC_LINKING, CMake’i statik bağlantıyı devre dışı bırakmada kullanmak için bir bayrak ve -j8 Automake’in derlemede 8 işlem kullanması için kullanılan bir bayraktır.

    cd qiskit-ibmq-provider

  3. Test veya programlama hataları için kontroller yürütmek istiyorsanız, geliştirici gereksinimlerini yükleyin. Kaynaktan kurulum yaparken qiskit-ibmq-provider paketini kurmak veya kullanmak için bu gerekli değildir.

    pip install -r requirements-dev.txt

  4. Qiskit-ibmq-provider’ı yükle.

    pip install .

Düzenlenebilir modda kurulum gerçekleştirmek istiyorsanız, projedeki kod değişikliklerinin yeniden yüklenmesi gerekmez:

pip install -e .

Qiskit Versioning¶

The Qiskit project is made up of several elements each performing different functionality. Each is independently useful and can be used on their own, but for convenience we provide this repository and meta-package to provide a single entrypoint to install all the elements at once. This is to simplify the install process and provide a unified interface to end users. However, because each Qiskit element has it’s own releases and versions some care is needed when dealing with versions between the different repositories. This document outlines the guidelines for dealing with versions and releases of both Qiskit elements and the meta-package.

For the rest of this guide the standard Semantic Versioning nomenclature will be used of: Major.Minor.Patch to refer to the different components of a version number. For example, if the version number was 0.7.1, then the major version is 0, the minor version 7, and the patch version 1.

Meta-package Version¶

The Qiskit meta-package version is an independent value that is determined by the releases of each of the elements being tracked. Each time we push a release to a tracked component (or add an element) the meta-package requirements, and version will need to be updated and a new release published. The timing should be coordinated with the release of elements to ensure that the meta-package releases track with element releases.

Adding New Tracked Elements¶

When a new Qiskit element is being added to the meta-package requirements, we need to increase the Minor version of the meta-package.

For example, if the meta-package is tracking 2 elements qiskit-aer and qiskit-terra and it’s version is 0.7.4. Then we release a new element qiskit-ignis that we intend to also have included in the meta-package. When we add the new element to the meta-package we increase the version to 0.8.0.

Patch Version Increases¶

When any Qiskit element that is being already tracked by the meta-package releases a patch version to fix bugs in a release we need also bump the requirement in the setup.py and then increase the patch version of the meta-package.

For example, if the meta-package is tracking 3 elements qiskit-terra==0.8.1, qiskit-aer==0.2.1, and qiskit-ignis==0.1.4 with the current version 0.9.6. When qiskit-terra release a new patch version to fix a bug 0.8.2 the meta-package will also need to increase it’s patch version and release, becoming 0.9.7.

Additionally, there are occasionally packaging or other bugs in the meta-package itself that need to be fixed by pushing new releases. When those are encountered we should increase the patch version to differentiate it from the broken release. Do not delete the broken or any old releases from pypi in any situation, instead just increase the patch version and upload a new release.

Minor Version Increases¶

Besides adding a new element to the meta-package the minor version of the meta-package should also be increased anytime a minor version is increased in a tracked element.

For example, if the meta-package is tracking 2 elements qiskit-terra==0.7.0 and qiskit-aer==0.1.1 and the current version is 0.7.5. When the qiskit-aer element releases 0.2.0 then we need to increase the meta-package version to be 0.8.0 to correspond to the new release.

Major Version Increases¶

The major version is different from the other version number components. Unlike the other version number components, which are updated in lock step with each tracked element, the major version is only increased when all tracked versions are bumped (at least before 1.0.0). Right now all the elements still have a major version number component of 0 and until each tracked element in the meta-repository is marked as stable by bumping the major version to be >=1 then the meta-package version should not increase the major version.

The behavior of the major version number component tracking after when all the elements are at >=1.0.0 has not been decided yet.

Optional Extras¶

In addition to the tracked elements there are additional packages built on top of Qiskit which are developed in tandem with Qiskit for example the application repositories like qiskit-optimization. For convienence these packages are tracked by the Qiskit metapackage as optional extras that can be installed with Qiskit. Releases of these optional downstream projects do not trigger a metapackage release as they are unpinned and do not effect the metapackage version. If there is a compatibility issue between Qiskit and these downstream optional dependencies and the minimum version needs to be adjusted in a standalone release this will only be done as a patch version release as it’s a packaging bugfix.

Qiskit Element Requirement Tracking¶

While not strictly related to the meta-package and Qiskit versioning how we track the element versions in the meta-package’s requirements list is important. Each element listed in the setup.py should be pinned to a single version. This means that each version of Qiskit should only install a single version for each tracked element. For example, the requirements list at any given point should look something like:

requirements = [
    "qiskit_terra==0.7.0",
    "qiskit-aer==0.1.1",
]

This is to aid in debugging, but also make tracking the versions across multiple elements more transparent.
