入门指南¶

安装¶

Qiskit 支持 Python 3.7 或更新版本。然而，Python 和 Qiskit 的生态都在不断演化，当其中有新版本出现时，可能会存在兼容性问题。

我们建议安装 Anaconda，一个用于科学计算的跨平台 Python 发行版，也建议使用包含在 Anaconda 中的 Jupyter 与 Qiskit 进行互动。

我们建议使用 Python 虚拟环境，将 Qiskit 与其他应用程序完全分开，从而改善您的体验。

使用虚拟环境最简单的方法是用 Anaconda 中的 conda 命令。Conda 环境允许您指定特定版本的 Python 和 Python 库。在您想要工作的目录中打开一个终端窗口。

建议您使用与 Anaconda 一起安装的 Anaconda prompt。您要做的是，在 Anaconda 中建立一个虚拟环境，并启动该环境。 无论是 Windows 还是 Linux 系统，这些命令都可以在 Anaconda prompt 中执行。

建立一个仅安装 Python 的最小环境。

conda create -n ENV_NAME python=3

激活您的新环境。

conda activate ENV_NAME

接下来，安装 Qiskit 包。

pip install qiskit

如果软件包已正确安装，那么您运行 conda list 可以看到虚拟环境中被链接的软件包。

如果您计划使用可视化功能或 Jupyter Notebook ，建议您将 Qiskit 与额外的 visualization 支持一同安装：

pip install qiskit[visualization]

需要注意的是，如果您是 zsh 用户（这是较新版本 macOS 上的 默认登录 Shell ），则需要将 qiskit[visualization] 加到引号中：

pip install 'qiskit[visualization]'

下列云端供应商已在其环境中预装 Qiskit：

IBM Quantum Lab

Build quantum applications and experiments with Qiskit in a cloud programming environment.

Strangeworks

A platform that enables users and organizations to easily apply quantum computing to their most pressing problems and research.

从源代码安装 Qiskit 的各组成部分，这将使您用到最新版本的 Qiskit，而不是 Python Package Index (PyPI) 仓库中的版本。这会让您更高效的检验与扩展最新版本的 Qiskit 代码。

从源代码安装 Qiskit 的各组成部分和组件时，会默认使用 development 版本（ 与 master git 分支相对应），而不是 stable 版本（与已发布的 pip 包相同的代码）。通常由于 development 的某个组成部分或组件包含了新的功能与变化，它们一般需要配合其他部分的 development 版本一同使用。

Terra 包和 Aer 包安装前都需要先用编译器从源代码构建。而 Ignis、Aqua 以及 IBM Quantum Provider 后端则不需要编译器。

从源代码安装 Qiskit 的各组成部分需要按照以下顺序进行，以防止 pip 版本低于源代码版本时，安装的各组成部分的版本低于预期：

  1. qiskit-terra

  2. qiskit-aer

  3. qiskit-ignis

  4. qiskit-ibmq-provider （如果您想连接到 IBM Quantum 设备或线上模拟器）

若要同时使用多个组件和组成部分，请针对每个 Qiskit 的组成部分使用以下步骤。

备注

由于在 Python 中使用命名空间打包，因此必须小心的安装软件包。 如果您计划从源文件安装任何 Qiskit 的组成部分，请不要使用 qiskit 元包。 此外，遵循本指南并使用单独的虚拟环境进行开发。 如果您确实需要在现有的安装环境进行开发，请参阅 https://github.com/pypa/sample-namespace-packages/blob/master/table.md，获取可以在一起兼容的组合的安装方法。

Set up the Virtual Development Environment

conda create -y -n QiskitDevenv python=3
conda activate QiskitDevenv

Installing Terra from Source

从源安装需要您在系统上有支持 C++11 的 C++ 编译器。

大多数 Linux 平台已安装了必需的 GCC 编译器。

如果您使用的是 MacOS，您可以通过安装 XCode来安装 Clang 编译器。 您可以打开终端窗口并输入以下内容，来检查是否安装了 XCode 和 Clang 。

clang --version

使用以下命令安装 XCode 和 Clang 。

xcode-select --install

在 Windows 上，最容易的方法是通过 Build Tools for Visual Studio 2019 安装 Visual C++ 编译器。当然您也可以通过 Visual Studio 2015 或 2017 来安装，安装时请确保勾选了安装 C++ 编译器的选项。

安装编译器后，就可以安装 Qiskit Terra 了。

  1. 克隆 Terra 仓库。

    git clone https://github.com/Qiskit/qiskit-terra.git

  2. 克隆代码仓库时，会创建名为 qiskit-terra 的本地文件夹。

    cd qiskit-terra

  3. 从 qiskit-terra 目录安装 Python 需求库。

    pip install cython

  4. 如果要执行测试或代码校验，请安装开发人员必需的构建库。

    pip install -r requirements-dev.txt

  5. 安装 qiskit-terra。

    pip install .

如果您想安装在可编辑模式下，即改变项目中的代码时不需要重装就能使用，您可以输入以下命令：

pip install -e .

然后，您可以在安装 Terra 后运行代码示例。 使用以下命令来运行示例。

python examples/python/using_qiskit_terra_level_0.py

备注

如果您不打算安装任何其他组件，qiskit-terra 将产生一个 RuntimeWarning 的警告，提醒 qiskit-aer 和 qiskit-ibmq-provider 没有被安装。 这样做是因为用户通常会使用其他 Qiskit 的组件，但并未意识到还没有安装它们， 或是由于某种原因安装 Aer 或 IBM Quantum Provider 的时候安装失败了。 如果您想要禁止这些警告，请添加：

import warnings
warnings.filterwarnings('ignore', category=RuntimeWarning,
                        module='qiskit')

在任何 qiskit 导入您的代码之前。 这将消除缺失 qiskit-aer 和 qiskit-ibmq-provider的警告，但仍将继续显示来自 qiskit 或其他软件包的任何其他警告。

Installing Aer from Source

  1. 克隆 Aer 仓库。

    git clone https://github.com/Qiskit/qiskit-aer

  2. 安装必需的构建库。

    pip install cmake scikit-build cython

在此之后，安装 Aer 的步骤取决于您正在使用的操作系统。 由于 Aer 是一个用 Python 接口编译的 C++ 程序，所以构建的 Aer 二进制并非 Python 的依赖程序，不能跨操作系统平台任意安装。

Linux

  1. 安装必需的编译器。

    构建 Aer 需要 C++ 编译器和开发的头文件。

    如果您正在使用 Fedora 或类似的 Linux 分发版，请使用:

    dnf install @development-tools

    对于 Ubuntu/Debian，请使用:

    apt-get install build-essential

  2. 安装 OpenBLAS 开发的头文件。

    如果您正在使用 Fedora 或类似的 Linux 分发版，请使用:

    dnf install openblas-devel

    对于 Ubuntu/Debian，请使用:

    apt-get install libopenblas-dev
macOS

  1. 安装依赖。

    要在 macOS 上使用 Clang 编译器，您需要安装额外的库以支持 OpenMP 。您可以使用 brew 来安装这个和其他的依赖。

    brew install libomp

  2. 然后安装一个 BLAS 的实现; OpenBLAS 是缺省选择。

    brew install openblas

    接下来，安装 Xcode Command Line Tools。

    xcode-select --install
Windows

在 Windows 上，您可以使用 Anaconda3 或 Miniconda3 来安装所有依赖。

  1. 安装必需的编译器。

    conda install --update-deps vs2017_win-64 vs2017_win-32 msvc_runtime

Qiskit Aer 是量子电路的高性能模拟器框架。 它提供 several backends ，以实现不同的模拟目标。

conda install --update-deps -c conda-forge -y openblas cmake

  1. 直接构建并安装 qiskit-aer

    如果您已经安装了 pip < 19.0.0，并且您的环境不需要自定义构建，请运行：

    cd qiskit-aer
pip install .

    这将同时构建二进制文件并安装 Aer。

    或者，如果您安装了一个较新的 pip ，或有一些订制的需求，您可以手动构建一个 Python wheel。

    cd qiskit-aer
python ./setup.py bdist_wheel

    如果需要在 wheel 构建期间设置一个订制的选项，请参阅 Custom options during wheel builds 。

    在您构建 Python wheel 后，它将被存储在 Aer 仓库中的 dist/ 目录中。具体的版本取决于您的版本

    cd dist
pip install qiskit_aer-*.whl

    输出 wheel 文件的具体文件名取决于开发中 Aer 的当前版本。

Custom options

Aer 的构建系统使用 scikit-build 在使用 Python 的接口构建时运行编译。 它充当 setuptools 的接口，调用 CMake ，并为本地系统编译二进制文件。

由于编译二进制文件的复杂性，您可能需要将选项传递给构建过程的某一部分。 传递变量的方式是：

python setup.py bdist_wheel [skbuild_opts] [-- [cmake_opts] [-- build_tool_opts]]

其中方括号 [] 中的元素是可选的，skbuild_opts， cmake_opts， build_tool_opts 可以用您自己的标注替换。 以下网址提供了 CMake 选项的列表：https://cmake.org/cmake/help/v3.6/manual/cmake.1.html#options。 例如，您可以运行以下内容：

python setup.py bdist_wheel -- -- -j8

这会将参数 -j8 传递到底层构建系统（在这里是 Automake ），告诉它您要使用 8 个进程并行构建。

例如， Linux 上这些参数常见的是指定 C++ 编译器的特定版本（通常，缺省值可能太旧）：

python setup.py bdist_wheel -- -DCMAKE_CXX_COMPILER=g++-7

这将告诉 CMake 在编译 Aer 时使用 g++-7 命令而不是缺省 g++ 。

取决于您的环境，此问题的另一个常见情况是，您可能需要指定平台名称并关闭静态链接。

python setup.py bdist_wheel --plat-name macosx-10.9-x86_64 \
-- -DSTATIC_LINKING=False -- -j8

这里 --plat-name 是一个 setuptools 的参数，用于指定软件包元数据的平台名称。 -DSTATIC_LINKING 是使用 CMake 时禁用静态链接的参数，-j8 是使用 Automake 时指定 8 个进程编译的参数。

  1. 克隆 Ignis 仓库。

    git clone https://github.com/Qiskit/qiskit-ignis.git

    pip install .

平台支持¶

Qiskit 力求支持尽可能多的平台，但由于可用的测试资源及平台可用性的限制，并非所有平台都可以支持。 对 Qiskit 的平台支持分为 3 层，每层的支持级别各有不同。 对于这些平台之外的平台， Qiskit 可能仍可安装，但未经过测试，您将不得不从源代码构建 Qiskit (和可能的 Qiskit 相关依赖) 。

此外， Qiskit 仅支持 CPython。 当前不支持使用其他 Python 解释器。

第 1 层¶

第 1 层支持的平台在开发过程的上游经过完全的测试，以确保任何提议的改变都能正常运行。 预编译的二进制文件被构建、测试和发布到 PyPI 作为发布过程的一部分。 由于所有依赖关系都在这些平台上可用，因此这些平台只需要一个可以正常运行的 Python 环境就可以安装。

第 1 层平台当前为:

  • Linux x86_64 (Python的发行版本与 “manylinux 2014 <https://www.python.org/dev/peps/pep-0599/>`__ 打包规范 ” 兼容的分发版)

  • macOS x86_64 (10.9 或更高版本)

  • Windows 64 位

第 2 层¶

在开发过程中，第 2 层平台不会在上游进行测试。 但是，预先编译的二进制文件将作为发行版过程的一部分构建，测试并发布到 PyPI ，并且这些包可以仅使用一个正常运行的Python环境安装。

第 2 层平台当前为:

  • Linux i686 (Python的发行版本与 “manylinux 2014 <https://www.python.org/dev/peps/pep-0599/>`__ 打包规范 ” 兼容的分发版 ) < 3.10

  • Windows 32 位的 Python < 3.10

  • Linux aarch64 (Python的发行版本与 manylinux 2014 打包规范兼容的分发版)

第 3 层¶

在开发过程中，第 3 层平台不会在上游进行测试。 预编译二进制文件构建并发布到 PyPI 作为释放过程的一部分，完全没有测试。 仅使用一个正常运行的Python环境可能无法安装它们，可能还需要 C/C++ 编译器或其他程序作为安装过程的一部分从源代码构建依赖关系。 对这些平台的支持只尽了最大的努力。

第 3 层平台当前为:

  • Linux i686 (Python的发行版本与 “manylinux 2014 <https://www.python.org/dev/peps/pep-0599/>`__ 打包规范 ” 兼容的分发版 ) >= 3.10

  • Windows 32 位的 Python >= 3.10

  • macOS arm64 (10.15 或更高版本)
